UD3 · B2 – Práctica guiada: auditoría de uso e acceso

Configuración básica de auditoría en Windows 11 e Ubuntu, xeración de evidencias e rexistro dos resultados nunha ficha sinxela.

Introdución

Módulo

MP0369 – Implantación de sistemas operativos.

Ciclo

Administración de sistemas informáticos en rede.

Nivel

1.º curso.

Autora

Silvia M. Peña Pérez.

Nesta práctica vas configurar e comprobar a auditoría básica de uso e acceso en dúas máquinas virtuais:

  • Windows 11 local, sen dominio.
  • Ubuntu local, con interface gráfica, sen dominio.

Ao rematar deberías ser quen de:

  • activar políticas básicas de auditoría en Windows e comprobar os eventos xerados;
  • configurar auditoría sobre un cartafol concreto en Windows;
  • provocar e identificar eventos de inicio de sesión correcto e fallido;
  • localizar e interpretar entradas de autenticación en Ubuntu;
  • comprobar como queda rexistrado o uso de sudo;
  • documentar o proceso nunha ficha sinxela.
Nota importante. Traballa sempre nas túas propias VMs. Podes consultar a un compañeiro/a se tes dúbidas, pero as capturas e a ficha final deben corresponder ás túas máquinas.

Recursos necesarios

  • VM con Windows 11 Pro, Education ou Enterprise.
  • VM con Ubuntu Desktop.
  • Usuario administrador local en Windows.
  • Usuario habitual en Ubuntu con permisos para executar sudo.
  • Este documento impreso ou en pantalla.
  • Ficha de rexistro incluída ao final deste documento.
Aviso sobre Windows. Esta práctica usa secpol.msc. Se o comando non abre, avisa á profesora. Algunhas edicións de Windows non inclúen a ferramenta de políticas de seguridade local.

Parte A – Auditoría en Windows 11

A1

Comprobación do estado inicial

  1. Inicia a túa VM con Windows 11 e inicia sesión cun usuario administrador local.
  2. Abre o Visor de eventos:
    • Preme Windows + R.
    • Escribe eventvwr.msc.
    • Preme Intro.
  3. No panel esquerdo, navega ata:
    Rexistros de Windows → Seguridade
  4. Observa o rexistro de seguridade:
    • Hai entradas?
    • Cantas aproximadamente?
    • Que tipos de eventos ves na columna Palabras clave?
    • Aparecen auditorías correctas, auditorías con erro ou ambas?
Anota na ficha. Número aproximado de entradas e tipos de eventos que observas neste momento.
A2

Activación das políticas de auditoría

  1. Abre o editor de políticas de seguridade local:
    • Preme Windows + R.
    • Escribe secpol.msc.
    • Preme Intro.
  2. Navega ata:
    Configuración de seguridade → Políticas locais → Política de auditoría
    Verás unha lista de categorías. O estado actual pode ser Sen auditoría nalgunhas delas.
  3. Activa as seguintes políticas facendo dobre clic en cada unha e marcando as opcións indicadas:
    Política Activar
    Auditar eventos de inicio de sesión Correcto e Erro
    Auditar o acceso a obxectos Correcto e Erro
    Auditar o uso de privilexios Erro
    Auditar eventos do sistema Correcto e Erro
  4. Pecha secpol.msc.
Captura. Fai unha captura da lista de políticas co novo estado activado.
Aclaración técnica. Activar a auditoría permite que Windows rexistre determinados sucesos, pero non significa que todos os accesos a todos os ficheiros queden automaticamente rexistrados. Para auditar un cartafol concreto tamén hai que configurar a súa auditoría desde as propiedades de seguridade do cartafol.
A3

Xeración de eventos de inicio de sesión

  1. Pecha a sesión actual do usuario administrador:
    Menú Inicio → icona do usuario → Pechar sesión
  2. Na pantalla de inicio de sesión, introduce intencionadamente unha contrasinal incorrecta para o usuario administrador. Faino tres veces seguidas.
  3. Introduce agora a contrasinal correcta e inicia sesión.
  4. Abre de novo:
    Visor de eventos → Rexistros de Windows → Seguridade
  5. Busca eventos recentes relacionados coa hora na que fixeches as probas. Deberías localizar eventos como estes:
    ID de evento Significado xeral
    4625 Inicio de sesión fallido
    4624 Inicio de sesión correcto
    É posible que Windows xere máis eventos arredor do mesmo momento. Usa a hora aproximada e o ID de evento para localizar os que correspondan coa túa proba.
  6. Fai clic nun evento 4625 e le a descrición completa no panel inferior.
Anota na ficha. Data/hora, ID, tipo e resumo da descrición de polo menos dous eventos.
Captura. Fai unha captura que mostre eventos 4625 e 4624 no rexistro.
A4

Auditoría sobre un cartafol

Nesta parte vas configurar a auditoría sobre un cartafol concreto e provocar un acceso denegado para que Windows xere un evento relacionado co acceso a obxectos.

Idea clave. Auditar un cartafol non cambia os permisos. Para xerar un evento de acceso denegado necesitas dúas cousas:
  1. Configurar auditoría sobre o cartafol.
  2. Facer que un usuario real non teña permiso para acceder ou escribir nel.
A4.1

Crear o cartafol de proba

  1. Crea o seguinte cartafol: 
    C:\Auditoria\UD3_Privado
    Se non existe a carpeta C:\Auditoria, créaa primeiro.
  2. Dentro de UD3_Privado, crea un ficheiro de texto chamado: 
    aviso.txt
    Escribe dentro unha frase sinxela, por exemplo: 
    Cartafol privado para a práctica de auditoría.
A4.2

Crear un segundo usuario local

  1. Crea un segundo usuario local sen privilexios administrativos:
    Configuración → Contas → Familia e outros usuarios → Engadir outro usuario
  2. Crea o usuario: 
    proba_audit
    Usa unha contrasinal temporal indicada pola profesora ou acordada para a práctica.
Non uses unha contrasinal persoal nin unha contrasinal que empregues noutros servizos.
A4.3

Configurar permisos NTFS para provocar acceso denegado

  1. Volle ao usuario administrador.
  2. Fai clic dereito sobre o cartafol: 
    C:\Auditoria\UD3_Privado
    Entra en:
    Propiedades → Seguridade → Avanzada
  3. Revisa que o usuario administrador ou o grupo Administradores teña control sobre o cartafol.
  4. Engade unha regra explícita para o usuario proba_audit:
    • Principal: proba_audit
    • Tipo: Denegar
    • Permisos básicos: Lectura e execución e Escritura
    Aplica os cambios.
Nun contorno real hai que ter coidado coas regras de denegación explícita, porque poden complicar a administración de permisos. Aquí úsanse só para provocar unha evidencia clara na práctica.
A4.4

Configurar a auditoría do cartafol

  1. Nas propiedades avanzadas do cartafol, entra na pestana:
    Auditoría
  2. Preme Engadir e configura unha entrada de auditoría:
    • Principal: proba_audit
    • Tipo: Erro
    • Aplicar a: Este cartafol, subcartafoles e ficheiros
    • Permisos básicos: Lectura e execución e Escritura
    Aplica os cambios e pecha as propiedades.
A4.5

Xerar o acceso denegado

  1. Cambia ao usuario proba_audit:
    Menú Inicio → icona do usuario → Cambiar usuario
  2. Tenta acceder ao cartafol: 
    C:\Auditoria\UD3_Privado
  3. Tenta tamén crear un ficheiro dentro do cartafol. O obxectivo é provocar unha mensaxe de acceso denegado.
  4. Volve á sesión do usuario administrador.
A4.6

Localizar o evento no Visor de eventos

  1. Abre:
    Visor de eventos → Rexistros de Windows → Seguridade
  2. Busca eventos arredor da hora na que fixeches a proba co usuario proba_audit. Podes empregar:
    Filtrar rexistro actual...
  3. Busca eventos relacionados co acceso a obxectos. Algúns IDs que poden aparecer son:
    ID de evento Posible significado
    4663 Intento de acceso a un obxecto
    4656 Solicitude de identificador para un obxecto
    4670 Cambio de permisos sobre un obxecto
  4. Abre o evento que pareza relacionado co cartafol UD3_Privado e revisa:
    • usuario;
    • data e hora;
    • nome do obxecto;
    • tipo de acceso;
    • se foi unha auditoría correcta ou unha auditoría con erro.
Anota na ficha. Se apareceu algún evento relacionado co cartafol, con que ID e que describe.
Captura. Fai unha captura do evento atopado.
Se non aparece ningún evento, indícao na ficha e explica que comprobacións farías: política de auditoría activada, entrada de auditoría no cartafol, permisos NTFS e hora exacta da proba.

Parte B – Auditoría e rexistros en Ubuntu

B1

Localización dos logs de autenticación

  1. Inicia a túa VM Ubuntu e inicia sesión co usuario habitual.
  2. Abre un terminal:
    Ctrl + Alt + T
  3. Consulta as últimas liñas do log de autenticación: 
    sudo tail -n 30 /var/log/auth.log
    Este comando mostra as últimas 30 liñas do log de autenticación.
  4. Observa as liñas que aparecen:
    • Que tipos de eventos ves?
    • Aparece o teu usuario?
    • Hai algún evento relacionado con sudo?
    • Hai algún evento relacionado co inicio ou peche de sesión?
Anota na ficha. 2 ou 3 liñas que che parezan relevantes e explica brevemente que indica cada unha.
Se /var/log/auth.log non existe ou non mostra información suficiente, usa journalctl, como se indica máis adiante nesta práctica.
B2

Xeración de eventos de autenticación

  1. Pecha a sesión gráfica actual:
    Menú do sistema → Pechar sesión
  2. Na pantalla de inicio de sesión, introduce intencionadamente unha contrasinal incorrecta dúas veces.
  3. Despois introduce a contrasinal correcta e inicia sesión.
  4. Abre un terminal e consulta de novo o log: 
    sudo tail -n 40 /var/log/auth.log
  5. Busca liñas relacionadas coa hora dos intentos fallidos e do inicio correcto. Segundo a configuración da VM, poden aparecer termos como: 
    authentication failure
gdm-password
pam_unix
session opened
session closed
    Se o acceso se fixese por SSH, poderían aparecer tamén mensaxes como: 
    Failed password
Accepted password
Anota na ficha. Unha liña relacionada cun intento fallido e unha liña relacionada cun inicio correcto ou apertura de sesión.
Captura. Fai unha captura do terminal mostrando esas liñas.
B3

Rexistro do uso de sudo

  1. Executa os seguintes comandos no terminal, un a un: 
    sudo apt update
sudo ls /root
  2. Consulta o log para ver como quedaron rexistrados: 
    sudo grep "sudo" /var/log/auth.log | tail -n 10
  3. Localiza as liñas correspondentes aos comandos que acabas de executar. Nunha liña de sudo adoita aparecer información como:
    • usuario que executou o comando;
    • terminal desde o que se executou;
    • directorio de traballo;
    • usuario de destino, normalmente root;
    • comando executado.
Anota na ficha. Que información aparece en cada liña.
Captura. Fai unha captura do terminal mostrando as liñas de sudo.
B4

Consulta con journalctl

  1. Usa journalctl para ver eventos recentes do sistema relacionados con autenticación, sesións ou sudo: 
    sudo journalctl --since "today" | grep -Ei "sudo|login|auth|authentication|session|gdm" | tail -n 30
  2. Se a túa VM ten SSH instalado e activo, tamén podes consultar: 
    sudo journalctl -u ssh --since "today"
    Se non tes SSH instalado ou activo, este segundo comando pode non devolver resultados útiles. Non pasa nada.
  3. Observa o formato das entradas e compárao co de /var/log/auth.log.
Anota na ficha. Unha diferenza que observes entre o formato de auth.log e o de journalctl.

Parte C – Reflexión e documentación

C1

Impacto no rendemento e tamaño dos rexistros

Responde na ficha:

  1. En Windows, activaches auditoría sobre inicios de sesión e acceso a un cartafol concreto. Que pasaría se activases a auditoría de todos os accesos correctos sobre todos os ficheiros do sistema? Que efecto tería no tamaño do rexistro e no rendemento?
  2. En Ubuntu, os logs de autenticación crecen co tempo. Que mecanismo existe para rotar, comprimir ou eliminar rexistros antigos e evitar que ocupen demasiado espazo?
Pista: en moitos sistemas Linux emprégase unha ferramenta chamada logrotate.
