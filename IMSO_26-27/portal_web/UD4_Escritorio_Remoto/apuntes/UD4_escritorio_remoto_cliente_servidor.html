1. Contexto: por que é importante o escritorio remoto

Nun contorno profesional, un técnico de sistemas non sempre está fisicamente diante dos equipos que ten que administrar. É habitual ter que administrar servidores, dar soporte a postos de traballo ou permitir teletraballo desde outras localizacións.

O escritorio remoto permite controlar un equipo á distancia coma se estivésemos sentados diante del: vemos o seu escritorio e usamos o teclado e o rato desde o equipo cliente, pero as aplicacións e os datos seguen executándose e gardándose no equipo remoto.

Exemplos de uso real:

  • Un técnico conecta remotamente cun servidor Windows para instalar actualizacións fóra do horario laboral, sen necesidade de desprazarse ao CPD.
  • Un empregado que teletraballa accede ao seu PC da oficina para usar aplicacións internas que só están dispoñibles dentro da rede corporativa, normalmente a través dunha VPN e dunha conexión de escritorio remoto.
  • Nun laboratorio de FP, o profesorado pode empregar ferramentas de acceso remoto para axudar, comprobar configuracións ou solucionar incidencias nos equipos do alumnado.

⚠️ Matiz importante: RDP non é exactamente o mesmo ca unha ferramenta de asistencia remota. RDP permite iniciar unha sesión remota nun equipo, pero non sempre serve para ver a mesma pantalla que está vendo o usuario local. Para soporte directo ao usuario poden empregarse ferramentas específicas de asistencia remota ou control compartido de pantalla.

2. Arquitectura cliente-servidor aplicada ao escritorio remoto

2.1. Modelo básico

O escritorio remoto é un exemplo claro do modelo cliente-servidor:

  • Servidor: equipo ao que accedemos remotamente. Nel execútanse as aplicacións, gárdanse os ficheiros e créase ou móstrase a sesión de escritorio.
  • Cliente: equipo desde o que nos conectamos. Envía pulsacións de teclado, movementos de rato e outras accións do usuario; recibe a representación gráfica do escritorio remoto.

É importante entender que:

  • O software e os datos seguen estando no servidor en todo momento.
  • O cliente non necesita ter a mesma potencia nin as mesmas aplicacións instaladas.
  • O cliente só precisa un programa de conexión compatible: cliente RDP, cliente VNC, Remmina, etc.

2.2. Fluxo simplificado dunha sesión

  1. O cliente abre o programa de conexión, por exemplo, Conexión a Escritorio remoto en Windows.
  2. Introduce o nome ou IP do servidor, o usuario e o contrasinal.
  3. O servidor comproba as credenciais.
  4. Se a autenticación é correcta e o usuario ten permiso, o servidor abre ou presenta unha sesión de escritorio.
  5. O servidor envía actualizacións gráficas ao cliente e recibe eventos de teclado e rato en tempo real.

💡 Idea clave: o ancho de banda dunha sesión RDP depende moito do movemento na pantalla, da resolución, da calidade gráfica e dos elementos redirixidos, como son, portapapeis, impresoras ou unidades locais. Non se envían ficheiros completos constantemente pola rede, senón actualizacións da sesión gráfica. Por iso o consumo adoita ser menor ca unha transmisión de vídeo equivalente, aínda que pode aumentar se hai moito movemento na pantalla.

2.3. Non todo acceso remoto é igual

Tecnoloxía Para que serve Interface gráfica Uso típico
RDP Escritorio remoto mediante o protocolo de Microsoft Si Administración gráfica de Windows, Windows Server ou xRDP en Linux
xRDP Servidor RDP para GNU/Linux Si Acceder a un escritorio Linux desde un cliente RDP
VNC Compartir ou controlar unha pantalla remota Si Soporte, aula, contornos mixtos, control da sesión visible
SSH Administración remota por terminal Non, salvo túneles ou X11 forwarding Administración de servidores Linux
Remmina Cliente gráfico de conexión remota Depende do protocolo Cliente Linux para RDP, VNC, SSH e outros protocolos

3. RDP en Windows e xRDP en Linux

3.1. Que é RDP (Remote Desktop Protocol)

RDP é un protocolo propietario de Microsoft para escritorio remoto. Permite que un cliente se conecte a un equipo remoto e controle unha sesión gráfica a distancia.

Características principais:

  • O porto por defecto é o 3389. Habitualmente identifícase co tráfico TCP 3389, aínda que nas versións actuais tamén pode empregarse UDP 3389 para mellorar o rendemento da sesión.
  • Pódese cambiar o porto por razóns de seguridade ou necesidades de configuración, aínda que cambiar o porto non substitúe outras medidas de protección.
  • Existen clientes RDP para Windows, Linux, macOS e outros sistemas.
  • O cliente e o servidor non teñen por que usar o mesmo sistema operativo, sempre que empreguen un protocolo compatible.

En Windows distínguense dous roles:

  • Equipo que acepta conexións remotas ou servidor RDP: require unha edición compatible, como Windows Pro, Enterprise, Education ou Windows Server. As edicións Home non permiten recibir conexións RDP entrantes de forma nativa.
  • Equipo que inicia conexións remotas ou cliente RDP: practicamente calquera edición de Windows, incluídas as edicións Home, pode actuar como cliente.

3.2. Activación do escritorio remoto en Windows

Pasos necesarios para activar o servidor RDP nun equipo Windows:

  1. Verificar que o equipo ten unha edición compatible: Pro, Enterprise, Education ou Server.
  2. Abrir Configuración → Sistema → Escritorio remoto.
  3. Activar a opción Permitir conexións de escritorio remoto a este equipo.
  4. Engadir os usuarios que poderán conectarse:
    • Os membros do grupo Administradores xa teñen permiso por defecto.
    • Outros usuarios deben engadirse ao grupo Usuarios de escritorio remoto.
  5. Comprobar o firewall:
    • A regra de Escritorio remoto debe permitir o tráfico de entrada necesario.
    • Como mínimo debe permitirse o porto TCP 3389 na rede correspondente.
    • En Windows modernos tamén pode existir tráfico UDP 3389 asociado á mellora de rendemento da sesión.
  6. Anotar a dirección IP ou o nome do equipo para que os clientes poidan conectarse.

⚠️ En contextos profesionais é imprescindible non expor directamente o porto 3389 a Internet. O acceso externo debe facerse mediante VPN, RD Gateway ou outro mecanismo de protección equivalente.

3.3. xRDP en Ubuntu: escritorio remoto con cliente RDP

En contornos mixtos pode ser útil que un equipo Linux acepte conexións desde clientes RDP de Windows. xRDP é un servidor de código aberto que implementa o protocolo RDP sobre un escritorio GNU/Linux.

Permite, por exemplo, que un usuario de Windows use o cliente Conexión a Escritorio remoto para acceder a un Ubuntu con contorno gráfico.

⚠️ Matiz: xRDP non é simplemente “activar o escritorio remoto de Ubuntu”. É un servizo adicional que permite conexións RDP contra unha sesión gráfica Linux. Ubuntu tamén pode empregar outras ferramentas de acceso remoto ou clientes como Remmina.

Instalación básica de xRDP en Ubuntu

  1. Actualizar a lista de paquetes e instalar xRDP:
BASH
sudo apt update
sudo apt install xrdp
  1. Activar o servizo e configuralo para que arranque co sistema:
BASH
sudo systemctl enable --now xrdp
sudo systemctl status xrdp
  1. Comprobar tamén o servizo asociado á xestión de sesións:
BASH
sudo systemctl status xrdp-sesman
  1. Se o servidor ten o firewall UFW activo, abrir o porto 3389 só para a rede do laboratorio. O rango debe adaptarse á rede real:
BASH
sudo ufw allow from 192.168.0.0/24 to any port 3389 proto tcp
sudo ufw status
  1. Desde o cliente Windows, abrir Conexión a Escritorio remoto, introducir a IP do servidor Ubuntu e autenticarse coas credenciais dun usuario existente no sistema.

Problema frecuente: pantalla negra en Ubuntu con GNOME

En Ubuntu 22.04/24.04 poden aparecer pantallas negras ou sesións que non cargan correctamente ao usar xRDP con GNOME. A incidencia pode depender do servidor gráfico, da sesión activa local, de Wayland/Xorg e da configuración concreta do escritorio.

Nun laboratorio educativo, unha solución práctica adoita ser instalar un escritorio lixeiro como XFCE e configurar xRDP para iniciar esa sesión.

  1. Instalar XFCE:
BASH
sudo apt install xfce4 xfce4-goodies
  1. Configurar a sesión remota para o usuario que se vai conectar por xRDP:
BASH
echo "xfce4-session" > ~/.xsession
  1. Reiniciar xRDP:
BASH
sudo systemctl restart xrdp

⚠️ Se hai varios usuarios que van conectarse por xRDP, cada usuario pode necesitar o seu propio ficheiro ~/.xsession.

⚠️ Tamén é recomendable probar cun usuario que non teña xa unha sesión gráfica local aberta. En determinados contornos, acceder por xRDP co mesmo usuario que xa iniciou sesión localmente pode provocar pantalla negra, peche de sesión ou comportamento irregular.

4. Seguridade en escritorio remoto

4.1. Principais riscos

RDP é un dos protocolos máis atacados cando está exposto a Internet. Os riscos máis habituais son:

  • Uso de contrasinais débiles en contas con acceso remoto.
  • Equipos accesibles desde calquera IP pública sen restrición.
  • Servidores ou clientes sen actualizar, vulnerables a fallos coñecidos.
  • Falta de rexistros e monitorización de accesos.
  • Uso de contas xenéricas ou compartidas, que impiden saber quen realizou unha acción.
  • Ausencia de bloqueo fronte a intentos repetidos de inicio de sesión.

4.2. Boas prácticas mínimas en rede interna

Nun laboratorio de FP ou nunha rede interna controlada, as medidas mínimas deberían ser:

  • Usar contrasinais robustos en todas as contas con acceso remoto.
  • Activar escritorio remoto só nos equipos onde sexa necesario.
  • Limitar os usuarios con acceso remoto.
  • Empregar contas persoais, non contas xenéricas compartidas.
  • Manter o sistema operativo e os compoñentes de RDP/xRDP actualizados.
  • Revisar periodicamente quen pertence ao grupo Usuarios de escritorio remoto.
  • Comprobar os rexistros cando haxa erros de acceso ou sospeita de uso indebido.

4.3. Boas prácticas en escenarios con acceso desde Internet

Nun escenario profesional, non se debe expor RDP directamente a Internet. O acceso externo debería facerse mediante unha capa previa de protección.

Medidas recomendables:

  • Usar unha VPN antes de permitir a conexión RDP.
  • Usar RD Gateway ou unha solución equivalente en contornos Microsoft cando sexa apropiado.
  • Activar NLA (Network Level Authentication) para que a autenticación se produza antes de crear a sesión completa.
  • Configurar bloqueo de conta tras varios intentos fallidos, tendo coidado de non provocar unha denegación de servizo accidental.
  • Considerar MFA para contas con acceso remoto.
  • Restrinxir o acceso a rangos de IP de confianza mediante firewall.
  • Auditar os inicios de sesión correctos e errados.
  • Manter todos os equipos actualizados.

💡 Pregunta tipo exame: por que a combinación VPN + RDP é máis segura que expor RDP directamente a Internet?
Porque a VPN engade unha capa de autenticación previa, cifra o tráfico e limita o acceso á rede interna. Un atacante externo non debería poder nin sequera tentar conectarse ao servizo RDP sen pasar antes pola VPN.

5. Diagnóstico de incidencias típicas

5.1. Problemas habituais

  1. O cliente non se pode conectar ao servidor
    • Servidor apagado ou desconectado da rede.
    • Servizo de escritorio remoto non activado no servidor.
    • Firewall bloqueando o porto 3389.
    • IP ou nome do servidor mal escrito no cliente.
    • Cliente e servidor en redes diferentes sen ruta entre elas.
  2. Usuario non autorizado
    • A conta non pertence ao grupo Usuarios de escritorio remoto nin é administradora.
    • Contrasinal incorrecto.
    • Conta bloqueada, caducada ou deshabilitada.
    • Políticas de seguridade que impiden o acceso.
  3. Conexión moi lenta ou que se corta
    • Problemas de ancho de banda ou latencia.
    • Configuración de calidade gráfica demasiado alta para o enlace dispoñible.
    • Conxestión doutros servizos na mesma rede.
    • Redirección de recursos locais innecesarios, como impresoras, unidades ou portapapeis.
  4. Problemas específicos de xRDP en Linux
    • Servizo xrdp parado.
    • Servizo xrdp-sesman parado.
    • Pantalla negra por incompatibilidade co escritorio gráfico.
    • Usuario xa iniciado nunha sesión gráfica local.
    • Ficheiro ~/.xsession inexistente ou mal configurado.
    • Firewall UFW bloqueando o porto.

5.2. Protocolo de diagnóstico

Ante un erro de conexión, convén seguir sempre unha orde lóxica:

Paso Que comprobar Como comprobalo
1. Rede Conectividade entre cliente e servidor ping <IP_servidor>
2. Configuración IP IP, máscara, pasarela e DNS Windows: ipconfig / Linux: ip a
3. Porto Se o porto 3389 responde Windows: Test-NetConnection <IP_servidor> -Port 3389
4. Servizo Escritorio remoto activo no servidor Windows: Configuración de Escritorio remoto / Linux: systemctl status xrdp
5. Sesión xRDP Xestión de sesións activa Linux: systemctl status xrdp-sesman
6. Firewall Porto 3389 permitido Firewall de Windows / Linux: sudo ufw status
7. Usuario Conta con permisos de acceso remoto Grupo Usuarios de escritorio remoto ou permisos equivalentes
8. Credenciais Contrasinal correcto e conta non bloqueada Proba de inicio de sesión e revisión de eventos ou logs
9. Rexistros Causa concreta do erro Windows: eventvwr / Linux: /var/log/xrdp*.log ou journalctl

5.3. Comandos útiles en Windows

No cliente ou servidor Windows:

POWERSHELL
hostname
ipconfig
ping <IP_servidor>
Test-NetConnection <IP_servidor> -Port 3389

No servidor Windows, para revisar regras de firewall relacionadas co Escritorio remoto:

POWERSHELL
Get-NetFirewallRule -DisplayGroup "Escritorio remoto"

Tamén se pode abrir o visor de eventos:

POWERSHELL
eventvwr

5.4. Comandos útiles en Ubuntu/xRDP

No servidor Ubuntu con xRDP:

BASH
systemctl status xrdp
systemctl status xrdp-sesman
ss -tulpn | grep 3389
sudo ufw status
journalctl -u xrdp
journalctl -u xrdp-sesman

Tamén poden revisarse os ficheiros de rexistro de xRDP:

BASH
ls -l /var/log/xrdp*.log
sudo tail -n 50 /var/log/xrdp.log
sudo tail -n 50 /var/log/xrdp-sesman.log

6. Documentación técnica da implantación

Saber activar e usar o escritorio remoto non é suficiente. Un técnico de sistemas debe ser capaz de documentar o que implantou, incluíndo probas, incidencias e accións de soporte.

Unha memoria técnica completa debería incluír:

  1. Introdución e obxectivo
    • Servizo implantado: RDP, xRDP ou outra solución.
    • Finalidade: administración, laboratorio, teletraballo, soporte, etc.
  2. Entorno de laboratorio
    • VMs ou equipos usados.
    • Sistema operativo de cada equipo.
    • RAM, disco, IP e rol: servidor ou cliente.
    • Esquema sinxelo de rede.
  3. Implantación do servidor
    • Pasos seguidos para activar RDP ou instalar xRDP.
    • Configuración de usuarios autorizados.
    • Porto empregado.
    • Configuración de firewall.
  4. Configuración dos clientes
    • Programa de conexión usado: mstsc, Remmina ou outro.
    • IP ou nome do servidor.
    • Usuario empregado.
    • Parámetros relevantes da conexión.
  5. Probas realizadas
    • Proba de conectividade.
    • Proba de conexión remota correcta.
    • Proba con usuario autorizado.
    • Proba con usuario non autorizado, se procede.
    • Proba de firewall ou porto.
  6. Incidencias e resolución
    • Síntoma observado.
    • Diagnóstico realizado.
    • Causa probable.
    • Solución aplicada.
    • Comprobación final.
  7. Asistencia remota documentada
    • Problema comunicado polo usuario.
    • Accións realizadas polo técnico.
    • Resultado final.
    • Recomendación ao usuario.
  8. Conclusións e melloras
    • Aprendizaxes principais.
    • Limitacións da configuración realizada.
    • Melloras que se aplicarían nun entorno real: VPN, restrición de IPs, MFA, auditoría, RD Gateway, etc.

6.1. Evidencias mínimas recomendadas

Nunha memoria de práctica deberían aparecer, como mínimo, estas evidencias:

Evidencia Que demostra
Captura da IP do servidor Que o alumno identifica correctamente o equipo remoto
Captura de RDP activado ou xrdp en execución Que o servizo está implantado
Captura da regra de firewall ou ufw status Que o porto necesario está permitido
Captura da conexión desde o cliente Que a conexión remota funciona
Captura do usuario conectado Que se empregou unha conta real e autorizada
Captura dunha proba de erro ou incidencia Que se realizou diagnóstico, non só instalación
Explicación da solución aplicada Que o alumno entende a causa e non só copia comandos

7. Conceptos clave

Termo Definición
RDP Remote Desktop Protocol. Protocolo propietario de Microsoft para escritorio remoto. Usa por defecto o porto 3389, principalmente TCP e, nas versións actuais, tamén pode usar UDP para mellorar o rendemento.
xRDP Servidor de código aberto que implementa o protocolo RDP sobre un escritorio GNU/Linux.
VNC Sistema de acceso remoto orientado a compartir ou controlar unha pantalla gráfica. Adoita usarse en soporte remoto e contornos mixtos.
SSH Protocolo seguro para administración remota por terminal, moi habitual en servidores Linux.
Remmina Cliente gráfico de conexión remota para GNU/Linux compatible con protocolos como RDP, VNC ou SSH.
NLA Network Level Authentication. Mecanismo que obriga a autenticarse antes de crear a sesión RDP completa, reducindo a superficie de ataque.
VPN Virtual Private Network. Túnel cifrado que conecta o equipo remoto coa rede interna. É unha medida esencial para acceso RDP desde Internet.
RD Gateway Servizo de contornos Microsoft que permite publicar acceso RDP de forma máis controlada, evitando expoñer directamente cada servidor ou PC.
Forza bruta Tipo de ataque que proba combinacións de usuario e contrasinal de forma masiva e automatizada. RDP exposto é un obxectivo habitual.
UFW Uncomplicated Firewall. Ferramenta de xestión do firewall en Ubuntu. Pode usarse para permitir o porto 3389 só desde a rede interna.
MFA Multi-Factor Authentication. Autenticación con dous ou máis factores. Recomendada para contas con acceso remoto en contornos profesionais.
