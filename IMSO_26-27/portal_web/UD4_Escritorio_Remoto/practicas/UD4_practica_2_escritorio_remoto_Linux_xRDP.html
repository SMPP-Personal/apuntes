Práctica 2 – Escritorio remoto en Linux con xRDP

Módulo: Implantación de Sistemas Operativos (IMSO) · 1.º ASIR

Unidade didáctica 4: Implantación de software específico en arquitectura cliente-servidor
**Duración estimada** 60–90 minutos
**Agrupamento** Parellas: un equipo fai de servidor Linux e outro de cliente Windows
**Entorno** Dúas VMs na mesma LAN ou rede virtual equivalente
**Nivel** Práctica guiada
**Tecnoloxía** xRDP — servidor RDP para GNU/Linux

1. Obxectivo da práctica

Configurar un servidor Ubuntu Desktop 22.04 ou 24.04 para aceptar conexións de escritorio remoto mediante o protocolo RDP usando xRDP, e conectarse a el desde un cliente Windows empregando o cliente estándar Conexión a Escritorio remoto (mstsc).

Ao rematar a práctica deberías ser capaz de:

  • identificar o equipo **servidor Linux** e o equipo **cliente Windows**;
  • comprobar a conectividade entre ambos equipos;
  • instalar e activar o servizo `xrdp`;
  • comprobar tamén o servizo `xrdp-sesman`;
  • configurar un escritorio lixeiro compatible, como **XFCE**;
  • abrir o porto 3389 só para a rede do laboratorio;
  • conectarte desde Windows usando `mstsc`;
  • diagnosticar erros habituais: firewall, servizo parado, usuario incorrecto, pantalla negra;
  • comparar configuracións de rendemento;
  • razoar medidas de seguridade para un contorno real.

2. Materiais e requisitos

2.1. Equipos necesarios

  • **Servidor Linux:**

VM con Ubuntu Desktop 22.04 ou 24.04 e contorno gráfico instalado.

  • **Cliente Windows:**

VM ou equipo físico con Windows 10/11.

  • Ambos equipos deben estar na **mesma LAN** ou nunha rede virtual equivalente.
  • O usuario do servidor Ubuntu debe ter permisos `sudo`.

⚠️ Esta práctica require un escritorio gráfico. Se empregas Ubuntu Server sen contorno gráfico, primeiro habería que instalar un escritorio, polo que non é o escenario recomendado para esta práctica inicial.

3. Antes de empezar: conceptos mínimos

3.1. Que é xRDP?

xRDP é un servidor de código aberto que permite aceptar conexións RDP nun sistema GNU/Linux.

Isto permite que un cliente Windows use o seu programa habitual:

Conexión a Escritorio remoto
mstsc

para conectarse a un escritorio Linux.

💡 **Idea clave:** xRDP non converte Ubuntu en Windows. O que fai é permitir que un cliente RDP acceda a unha sesión gráfica remota de Linux.

3.2. Que compoñentes imos usar?

Compoñente Función
**xrdp** Servizo principal que escoita conexións RDP no porto 3389
**xrdp-sesman** Xestor de sesións; encárgase de iniciar e controlar as sesións gráficas
**XFCE** Escritorio lixeiro que adoita funcionar mellor con xRDP ca GNOME en laboratorio
**UFW** Firewall sinxelo de Ubuntu
**mstsc** Cliente RDP integrado en Windows
**Xorg** Tipo de sesión que seleccionaremos na pantalla de login de xRDP

3.3. Por que instalamos XFCE?

En Ubuntu 22.04/24.04 con GNOME pode aparecer unha pantalla negra ou unha sesión que non carga correctamente ao conectarse por xRDP.

Isto pode depender de varios factores:

  • GNOME;
  • Wayland/Xorg;
  • sesión gráfica xa aberta localmente;
  • usuario usado para a conexión;
  • configuración de `.xsession`;
  • versión concreta dos paquetes.

Nun laboratorio educativo, unha solución práctica e estable adoita ser instalar XFCE e configurar a sesión remota para usalo.

⚠️ Evita conectar por xRDP co mesmo usuario que xa ten unha sesión gráfica local aberta na VM. Pode producir pantalla negra, peche inmediato da sesión ou comportamento irregular.

ping <IP_SERVIDOR>

Exemplo:

De volta no servidor Ubuntu, actualiza a lista de paquetes:

sudo apt update

Isto non actualiza todo o sistema; só descarga información actualizada dos repositorios.

Evidencia preliminar: IP do servidor, resultado do ping e versión de Ubuntu.

5. Parte 2 – Crear ou preparar o usuario para xRDP

5.1. Usuario recomendado para a práctica

Podes usar un usuario existente, por exemplo alumno, pero é recomendable usar un usuario específico para a sesión remota.

Neste documento usaremos como exemplo:

rdpuser

Se decides usar outro usuario, substitúe rdpuser polo nome real.

5.2. Crear usuario `rdpuser`

No servidor Ubuntu:

sudo adduser rdpuser

O sistema pedirá un contrasinal e algúns datos opcionais.

Recomendacións:

  • usa un contrasinal longo;
  • non uses contrasinais triviais como `1234`, `abc123`, `alumno` ou `password`;
  • anota o contrasinal só se o indica o docente e nunca nun documento público.

Completa:

Campo Valor
Usuario creado para xRDP rdpuser
O usuario ten contrasinal? Si / Non
Usarase este usuario para a conexión remota? Si / Non

⚠️ Non inicies sesión localmente na VM Ubuntu con `rdpuser` antes de probar xRDP. É mellor que ese usuario non teña unha sesión gráfica local aberta.

6. Parte 3 – Instalación e activación de xRDP

6.1. Instalar xRDP

No servidor Ubuntu:

sudo apt install xrdp

Se o sistema pregunta se queres continuar, responde:

S

ou:

Y

segundo o idioma da instalación.

6.2. Activar e iniciar o servizo

Executa estes comandos un a un:

sudo systemctl enable xrdp
sudo systemctl start xrdp
sudo systemctl status xrdp

O resultado esperado é ver:

active (running)

Se aparece outro estado, non continúes sen anotar o erro.

6.3. Comprobar `xrdp-sesman`

Agora comproba tamén o xestor de sesións:

sudo systemctl status xrdp-sesman

Tamén debería aparecer:

active (running)

💡 `xrdp-sesman` é importante porque xestiona as sesións gráficas. Se o login falla, se pecha a sesión ou aparece pantalla negra, este servizo e os seus logs son fundamentais para diagnosticar.

6.4. Comprobar que xRDP escoita no porto 3389

Executa:

sudo ss -tulnp | grep 3389

Deberías ver unha liña onde apareza xrdp escoitando no porto 3389.

Completa:

Comprobación Resultado
systemctl status xrdp active / non active
systemctl status xrdp-sesman active / non active
`ss -tulnp \ grep 3389` mostra xRDP? Si / Non

Evidencia 1: captura de systemctl status xrdp, systemctl status xrdp-sesman e ss -tulnp | grep 3389.

7. Parte 4 – Configuración de XFCE para a sesión remota

7.1. Instalar XFCE

Instala XFCE e complementos básicos:

sudo apt install xfce4 xfce4-goodies

Esta instalación pode tardar uns minutos.

7.2. Configurar `.xsession` para o usuario actual

O ficheiro .xsession indica que escritorio debe iniciar a sesión remota.

Se vas conectarte co mesmo usuario co que estás executando a terminal, usa:

echo xfce4-session > ~/.xsession
cat ~/.xsession

O resultado de cat ~/.xsession debería ser:

xfce4-session

7.3. Configurar `.xsession` para `rdpuser`

Se creaches o usuario rdpuser, configura tamén a súa sesión.

O xeito máis sinxelo é iniciar sesión con ese usuario nunha terminal:

su - rdpuser
echo xfce4-session > ~/.xsession
cat ~/.xsession
exit

Tamén se pode facer desde o usuario con sudo, pero para esta práctica é máis claro usar su - rdpuser.

⚠️ O ficheiro `.xsession` é por usuario. Se configuras `.xsession` no teu usuario actual, pero logo te conectas por xRDP con outro usuario, a configuración pode non aplicarse a ese outro usuario.

7.4. Reiniciar xRDP

Despois de instalar XFCE e configurar .xsession, reinicia xRDP:

sudo systemctl restart xrdp
sudo systemctl status xrdp
sudo systemctl status xrdp-sesman

Completa:

Comprobación Resultado
XFCE instalado Si / Non
.xsession configurado no usuario usado para xRDP Si / Non
xRDP reiniciado Si / Non

8. Parte 5 – Configuración básica do firewall

8.1. Comprobar estado de UFW

Antes de abrir portos, comproba se UFW está activo:

sudo ufw status verbose

Resultado posible:

Status: inactive

ou:

Status: active

Anota:

Campo Valor
UFW está activo? Si / Non
Política por defecto
Hai regras previas?

⚠️ Se UFW aparece como `inactive`, a regra que engadas non filtrará tráfico ata que UFW se active. Nun laboratorio pode manterse inactivo se así o indica o docente, pero hai que documentalo.

8.2. Abrir o porto 3389 só para a rede do laboratorio

Non abras xRDP a calquera IP se podes evitalo. Debes permitir só a rede do laboratorio.

Formato xeral:

sudo ufw allow from <REDE_LABORATORIO>/24 to any port 3389 proto tcp

Exemplo para unha rede 192.168.56.0/24:

sudo ufw allow from 192.168.56.0/24 to any port 3389 proto tcp

Se a túa rede é outra, adapta o comando. Por exemplo, se a IP do servidor é 192.168.1.50, probablemente a rede sexa:

192.168.1.0/24

E o comando sería:

sudo ufw allow from 192.168.1.0/24 to any port 3389 proto tcp

8.3. Comprobar a regra

Despois de engadir a regra:

sudo ufw status verbose

Debe verse unha regra para o porto 3389/TCP restrinxida á rede indicada.

Evidencia 2: captura de sudo ufw status verbose ou explicación de que UFW está inactivo por decisión do docente.

9. Parte 6 – Comprobación desde Windows antes de conectar

Antes de abrir mstsc, comproba desde Windows que o porto TCP 3389 responde.

No cliente Windows abre PowerShell:

Test-NetConnection -ComputerName <IP_SERVIDOR> -Port 3389

Exemplo:

Test-NetConnection -ComputerName 192.168.56.20 -Port 3389

Resultado esperado:

TcpTestSucceeded : True

Completa:

Campo Valor
IP do servidor usada
TcpTestSucceeded True / False
Se fallou, que revisarías?

Se aparece False, revisa:

  1. IP do servidor.
  2. Servizo `xrdp`.
  3. Servizo `xrdp-sesman`.
  4. Firewall UFW.
  5. Rede entre cliente e servidor.
  6. Perfil ou modo de rede da VM.

Evidencia 3: captura de Test-NetConnection.

11. Parte 8 – Probas de rendemento

11.1. Obxectivo

Comparar a sensación de uso con diferentes opcións gráficas do cliente RDP.

11.2. Conexión A: calidade alta

No cliente RDP, antes de conectar:

  1. Abre `mstsc`.
  2. Preme **Mostrar opcións**.
  3. Revisa a pestana **Experiencia**.
  4. Activa opcións de calidade alta se están dispoñibles.

Exemplos:

  • fondo de escritorio;
  • suavizado de fontes;
  • animacións;
  • temas;
  • redirección de audio.

11.3. Conexión B: calidade baixa

Repite a conexión, pero con opcións máis lixeiras:

  • sen fondo de escritorio;
  • sen animacións;
  • sen redirección de audio;
  • menos efectos visuais;
  • configuración orientada a conexión máis lenta.

11.4. Proba comparativa

En cada conexión realiza a mesma acción:

  • abrir dúas ou tres aplicacións;
  • mover xanelas;
  • escribir texto;
  • abrir o xestor de ficheiros;
  • cambiar entre xanelas.

Completa a táboa:

Fluidez ao mover xanelas

12. Parte 9 – Reflexión sobre seguridade

Responde por escrito en 8–10 liñas.

Preguntas obrigatorias

  1. Que risco tería activar xRDP nun servidor Ubuntu exposto directamente a Internet sen firewall nin VPN?
  2. Por que é máis seguro permitir xRDP só desde a LAN ou desde unha VPN?
  3. Que medidas aplicarías nun servidor Linux en produción?
  4. Para que serviría `fail2ban`?
  5. Por que `fail2ban` non substitúe unha VPN nin unha boa política de firewall?
  6. Que risco poden ter o portapapeis, a redirección de unidades, o audio ou as impresoras nunha sesión remota?
  7. Por que convén usar usuarios individuais e non contas compartidas?

12.1. Medidas recomendables nun servidor real

Nun servidor Linux en produción consideraríanse, como mínimo:

  • VPN antes de permitir acceso remoto;
  • firewall restrinxido a IPs ou redes autorizadas;
  • usuarios individuais;
  • contrasinais longos e non reutilizados;
  • actualizacións de seguridade;
  • revisión de logs;
  • `fail2ban` como medida complementaria;
  • desactivación de recursos redirixidos se non son necesarios;
  • auditoría de accesos;
  • MFA se a infraestrutura o permite.

12.2. Que é fail2ban?

fail2ban é unha ferramenta de Linux que monitoriza logs do sistema en busca de intentos repetidos de autenticación fallida.

Cando detecta demasiados intentos desde unha mesma IP, pode bloqueala temporalmente mediante regras de firewall.

É especialmente útil para protexer servizos como:

  • SSH;
  • xRDP;
  • outros servizos con autenticación.

⚠️ `fail2ban` é unha capa complementaria. Non substitúe VPN, restrición de IPs, actualizacións, contrasinais robustos nin unha configuración segura do servizo.
