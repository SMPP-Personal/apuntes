Práctica 1 – Escritorio remoto en Windows (RDP)

Módulo: Implantación de Sistemas Operativos (IMSO) · 1.º ASIR

Unidade didáctica 4: Implantación de software específico en arquitectura cliente-servidor
Duración estimada 60–90 minutos
Agrupamento Parellas: un equipo fai de servidor e outro de cliente
Entorno Dúas VMs ou dous equipos físicos na mesma LAN
Nivel Práctica guiada
Tecnoloxía RDP — Remote Desktop Protocol

1. Obxectivo da práctica

Configurar un equipo Windows nunha rede de laboratorio para aceptar conexións de escritorio remoto mediante RDP desde outro equipo cliente, aplicando medidas básicas de seguridade e documentando o proceso.

Ao rematar a práctica deberías ser capaz de:

  • diferenciar o equipo cliente RDP e o equipo servidor RDP;
  • comprobar a conectividade básica entre dous equipos;
  • activar Escritorio remoto nun Windows compatible;
  • autorizar un usuario concreto para conectarse por RDP;
  • comprobar o firewall e o porto 3389;
  • conectarte desde outro equipo usando mstsc;
  • documentar evidencias e incidencias;
  • explicar por que RDP non debe expoñerse directamente a Internet.

2. Materiais e requisitos

2.1. Equipos necesarios

  • Servidor RDP:

VM ou equipo físico con Windows 10/11 Pro, Enterprise ou Education, ou Windows Server.

  • Cliente RDP:

VM ou equipo físico con Windows 10/11. Para actuar como cliente serve tamén Windows Home.

  • Ambos equipos deben estar na mesma LAN ou nunha rede virtual equivalente.
  • Necesitas unha conta con privilexios de administración no equipo servidor.

⚠️ Recorda: as edicións Home de Windows 10/11 non permiten recibir conexións RDP de forma nativa. Poden actuar como cliente, pero non como servidor RDP.

3. Antes de empezar: conceptos mínimos

3.1. Que é RDP?

RDP (Remote Desktop Protocol) é o protocolo de escritorio remoto usado en Windows. Permite conectarse desde un equipo cliente a outro equipo servidor e manexar a súa sesión gráfica.

Nunha conexión RDP:

  • as aplicacións execútanse no servidor;
  • os ficheiros quedan no servidor;
  • o cliente recibe a imaxe da sesión remota;
  • o cliente envía teclado e rato;
  • poden redirixirse recursos como portapapeis, impresoras ou unidades locais, se a configuración o permite.

3.2. Porto usado por RDP

RDP usa por defecto o porto 3389.

Nesta práctica imos comprobar TCP 3389, que é a proba básica de conectividade RDP:

Test-NetConnection -ComputerName <IP_SERVIDOR> -Port 3389

Nas versións actuais de Windows, RDP tamén pode empregar UDP 3389 para mellorar rendemento e latencia, pero para diagnosticar se o servizo responde adoita comezarse comprobando TCP 3389.

3.3. Que é NLA?

NLA (Network Level Authentication) é a autenticación a nivel de rede. Obriga a que o usuario se autentique antes de crear a sesión RDP completa.

Debe deixarse activada sempre que sexa posible porque:

  • reduce a superficie de ataque;
  • evita crear sesións completas para usuarios non autenticados;
  • mellora a seguridade fronte a intentos masivos de conexión.

⚠️ Se o servidor é Windows Home, non continúes con ese equipo como servidor RDP. Terás que cambiar a unha VM ou equipo con Windows Pro, Enterprise, Education ou Server.

4.3. Comprobar IPs

No servidor e no cliente abre PowerShell ou CMD e executa:

ipconfig

Anota os datos:

Equipo Dirección IP Máscara Porta de enlace
Servidor
Cliente

4.4. Comprobar conectividade con ping

Desde o cliente, comproba se responde o servidor:

ping <IP_SERVIDOR>

Desde o servidor, comproba se responde o cliente:

ping <IP_CLIENTE>

Anota o resultado:

Proba Resultado
Cliente → Servidor Responde / Non responde
Servidor → Cliente Responde / Non responde

⚠️ Se o ping falla, non conclúas automaticamente que non hai rede. Windows pode bloquear ICMP polo firewall. Máis adiante comprobarás tamén o porto RDP con Test-NetConnection.

4.5. Crear o usuario remoto1

No servidor crea un usuario local chamado:

remoto1

Debe ter un contrasinal longo e non trivial:

  • preferiblemente 12 ou máis caracteres;
  • non reutilizado noutros servizos;
  • sen patróns evidentes como 1234, password, abc, nome do alumno ou nome do centro;
  • pode combinar palabras, números e símbolos.

Podes crealo desde a interface gráfica:

Configuración → Contas → Outros usuarios → Engadir conta

Ou desde administración de usuarios locais, se está dispoñible.

Cobre a ficha:

Campo Valor
Usuario creado remoto1
Tipo de conta Local
É administrador? Si / Non
Observacións

💡 Para esta práctica, o usuario remoto1 non precisa ser administrador. O máis seguro é que sexa usuario estándar e que só teña permiso para conectarse por RDP.

5. Parte 2 – Activar Escritorio remoto no servidor

5.1. Abrir a configuración de Escritorio remoto

No servidor:

Configuración → Sistema → Escritorio remoto

En versións antigas ou en Windows Server tamén podes atopalo en:

Panel de control → Sistema → Configuración de acceso remoto → Remoto

Activa:

Habilitar Escritorio remoto

Deixa marcada a opción de Autenticación a nivel de rede (NLA) se aparece.

5.2. Autorizar o usuario remoto1

Na configuración de Escritorio remoto:

  1. Fai clic en Seleccionar usuarios que poden acceder de forma remota a este equipo.
  2. Preme Engadir.
  3. Escribe:
remoto1
  1. Preme Comprobar nomes.
  2. Acepta.

Se aparece co nome do equipo diante, por exemplo:

NOME_SERVIDOR
emoto1

é correcto.

💡 Os membros do grupo Administradores adoitan ter acceso remoto por defecto, pero nesta práctica queremos comprobar explicitamente o acceso dun usuario concreto.

5.3. Revisar o firewall

Abre:

Firewall de Windows Defender con seguridade avanzada

Busca a regra de entrada:

Escritorio remoto (TCP de entrada)

Comproba:

  • que está habilitada;
  • que corresponde ao perfil de rede correcto;
  • que o equipo está nun perfil adecuado para o laboratorio.

Perfís posibles:

Perfil Uso típico
Dominio Equipo unido a dominio
Privado Rede de confianza, laboratorio ou casa
Público Rede non confiable

⚠️ Nun laboratorio controlado adoita ser preferible traballar con perfil Privado ou Dominio, non Público.

5.4. Comprobar o porto 3389 desde o cliente

No cliente abre PowerShell e executa:

Test-NetConnection -ComputerName <IP_SERVIDOR> -Port 3389

Debes substituír <IP_SERVIDOR> pola IP real do servidor.

Exemplo:

Test-NetConnection -ComputerName 192.168.1.50 -Port 3389

Fíxate especialmente neste campo:

TcpTestSucceeded : True

Resultado esperado:

Campo Significado
TcpTestSucceeded : True O porto TCP 3389 responde
TcpTestSucceeded : False O porto non responde ou está bloqueado

Se falla, revisa:

  1. O servidor está aceso?
  2. A IP é correcta?
  3. Escritorio remoto está activado?
  4. O firewall permite RDP?
  5. A rede está configurada correctamente?
  6. Estás no perfil de rede correcto?

💡 Se tes Telnet instalado, tamén poderías probar telnet <IP_SERVIDOR> 3389, pero Test-NetConnection está dispoñible en Windows 10/11 sen instalar compoñentes adicionais.

6.4. Aviso de certificado ou identidade

É posible que apareza unha mensaxe avisando de que non se pode verificar a identidade do equipo remoto.

Nun laboratorio podes aceptala se estás seguro de que a IP corresponde ao teu servidor.

Nun contorno real non se debe aceptar sen revisar:

  • nome do equipo;
  • certificado;
  • procedencia da conexión;
  • política da organización.

Anota se apareceu o aviso:

Pregunta Resposta
Apareceu aviso de certificado/identidade? Si / Non
Que fixeches?
Por que pode aparecer este aviso?

6.5. Proba funcional dentro da sesión remota

Unha vez dentro da sesión remota, comproba que estás traballando no servidor e non no cliente.

Dentro da sesión RDP:

  1. Abre o Explorador de ficheiros.
  2. Vai á carpeta Documentos do usuario remoto1.
  3. Crea un cartafol chamado:
Remoto
  1. Dentro crea un ficheiro de texto chamado:
proba_rdp.txt
  1. Escribe dentro unha frase breve, por exemplo:
Esta proba foi creada dentro da sesión remota RDP.
  1. Pecha sesión.
  2. No servidor, comproba que o cartafol e o ficheiro seguen existindo no perfil do usuario remoto1.

⚠️ Non crees o cartafol directamente en C:\, porque un usuario estándar pode non ter permisos para escribir aí. Usa Documentos ou Escritorio do propio usuario.

7. Parte 4 – Reflexión sobre seguridade

7.1. Pechar correctamente a sesión

Ao rematar a proba, pecha a sesión remota correctamente:

Inicio → Usuario → Pechar sesión

Non deixes a sesión aberta sen necesidade.

7.2. Cambiar o contrasinal

Cambia o contrasinal de remoto1 por outro tamén forte.

Non uses contrasinais débiles nin sequera para probar.

⚠️ Non crees nin engadas ao grupo RDP usuarios con contrasinais débiles no equipo do laboratorio. Basta con razoar por escrito que risco suporía.

7.3. Preguntas de reflexión

Responde por escrito en 8–10 liñas:

  1. Que riscos tería este servidor se estivese exposto directamente a Internet co porto 3389 aberto?
  2. Por que é máis seguro usar VPN + RDP que RDP directo desde Internet?
  3. Que función ten NLA?
  4. Que tres medidas adicionais aplicarías nunha empresa real?
  5. Por que non é boa idea usar contas compartidas para acceso remoto?

Podes mencionar:

  • VPN;
  • bloqueo temporal por intentos fallidos;
  • NLA;
  • MFA ou segundo factor;
  • restrición por IP;
  • actualizacións de seguridade;
  • logs e auditoría;
  • contrasinais longos e non reutilizados;
  • usuarios individuais;
  • permisos mínimos.
