Introdución ás GPO en dominios Windows

Como usar obxectos de directiva de grupo para configurar de forma centralizada usuarios e equipos dun dominio Windows.

Idea central: unha GPO créase, configúrase e despois lígase a un sitio, dominio ou OU. Os grupos poden axudar a filtrar, pero non son o punto principal de ligazón.

1. Punto de partida

O dominio xa existe; agora configuramos usuarios e equipos centralizadamente.

Neste bloque introducimos as Group Policy Objects (GPO) nun dominio Windows, aplicadas ao escenario da UD2.

2. Que é unha GPO

Unha GPO (Group Policy Object) é un obxecto de Active Directory que contén configuracións administrativas para usuarios e equipos.

Usuarios

  • restricións do escritorio;
  • configuración do menú Inicio;
  • scripts de inicio de sesión;
  • mapeo de unidades;
  • preferencias de usuario.

Equipos

  • configuracións de seguridade;
  • scripts de arranque ou apagado;
  • configuracións de rede;
  • firewall e sistema;
  • instalación ou configuración de software.
En vez de configurar cada equipo ou cada usuario manualmente, créase unha GPO e aplícase de forma centralizada desde Active Directory.

3. Unha GPO non se aplica directamente a un grupo

Ligazón non é o mesmo que filtrado.

Regra clave: unha GPO non se vincula directamente a un grupo como mecanismo principal.

As GPO vincúlanse a:

S
Sitios
Sites
D
Dominios
Domains
OU
Unidades organizativas
Organizational Units

Despois, se é necesario, unha GPO pode filtrarse mediante permisos ou grupos de seguridade. Pero iso é un filtro adicional, non a forma principal de aplicación.

ConceptoFunción principal
OUOrganizar usuarios/equipos e servir como punto de aplicación de GPO
GrupoAgrupar contas para permisos, dereitos ou filtrado
GPOAplicar configuracións a usuarios ou equipos
Security FilteringLimitar que usuarios/equipos reciben unha GPO xa vinculada
Incorrecto:
Crear unha GPO e “aplicala ao grupo GG_Alumnado_Curso1”.
Correcto:
Crear unha GPO, ligala á OU Curso1 e, se fai falta, filtrala para membros de GG_Alumnado_Curso1.

4. GPMC: consola de xestión de directivas de grupo

Para xestionar GPO en Windows Server emprégase a consola Group Policy Management Console (GPMC).

Como abrila

  • Server Manager → Tools → Group Policy Management.
  • Executando:
gpmc.msc

Que permite facer

  • ver dominio e OU;
  • crear e editar GPO;
  • ligar GPO a dominios ou OU;
  • modificar a orde de aplicación;
  • facer copias de seguridade;
  • consultar resultados de políticas.

5. Crear unha GPO non é o mesmo que ligala

Un erro frecuente é pensar que unha GPO se aplica só por estar creada. En realidade hai dous pasos diferentes.

1

Crear a GPO

Créase dentro do contedor Group Policy Objects. Aínda non se aplica a ninguén se non está ligada.

2

Ligar a GPO

Vincúlase a un dominio, sitio ou OU. A partir dese momento pode aplicarse aos usuarios ou equipos do ámbito.

Créase: GPO_Alumnado_Curso1_Escritorio Lígase: OU=Curso1 Resultado: aplícase aos usuarios/equipos desa OU segundo configuracións e filtros

Se unha GPO existe pero non está ligada, é como unha regra gardada pero non aplicada.

7. Orde de aplicación: LSDOU

As GPO seguen unha orde de aplicación coñecida como LSDOU.

L

D

cifpsauce.local └── OU=Alumnado └── OU=Curso1
  1. Política local do equipo.
  2. GPO do sitio, se existe.
  3. GPO do dominio cifpsauce.local.
  4. GPO da OU Alumnado.
  5. GPO da OU Curso1.
Como regra xeral, se varias GPO configuran o mesmo axuste, a configuración aplicada máis tarde na orde LSDOU pode prevalecer. Pero tamén inflúen a orde das ligazóns, Enforced, Block inheritance, filtros de seguridade, filtros WMI e se a configuración é de usuario ou equipo.

8. Computer Configuration e User Configuration

8.1. Computer Configuration

Aplica configuracións aos equipos.

  • no arranque do sistema;
  • periodicamente en segundo plano;
  • ao forzar actualización con gpupdate;
  • seguridade do equipo, firewall, scripts, software, restricións de sistema.

8.2. User Configuration

Aplica configuracións aos usuarios.

  • no inicio de sesión;
  • periodicamente en segundo plano;
  • ao forzar actualización con gpupdate;
  • escritorio, menú Inicio, scripts, unidades de rede, redirección e preferencias.

8.3. Erro moi común

Se configuras...A GPO debe aplicarse a...
Axustes de equipoOU onde están as contas de equipo
Axustes de usuarioOU onde están as contas de usuario
Para o nivel desta UD, a regra práctica será: Configuración de usuario → OU con usuarios. Configuración de equipo → OU con equipos.

9. Policies e Preferences

Policies

Son configuracións administradas.

  • impoñen unha configuración;
  • adoitan impedir ou revertir cambios do usuario;
  • úsanse para restricións e seguridade;
  • teñen carácter máis obrigatorio.
Preferences

Permiten distribuír configuracións máis flexibles.

  • mapear unidades de rede;
  • crear accesos directos;
  • configurar impresoras;
  • copiar ficheiros;
  • aplicar valores iniciais ou condicións.
Para mapear unha unidade de rede do profesorado, adoita ser máis cómodo usar Group Policy Preferences que un script manual de inicio de sesión.

10. Exemplos frecuentes de uso de GPO

A política xeral de contrasinais e bloqueo de contas dun dominio debe tratarse con coidado. Nunha práctica inicial pode revisarse ou axustarse desde a Default Domain Policy, lembrando que afecta ao dominio, debe documentarse e non é unha GPO calquera para probar sen consecuencias.

  • lonxitude mínima;
  • complexidade;
  • historial de contrasinais;
  • idade máxima ou mínima;
  • bloqueo tras intentos fallidos.

En contornos máis avanzados existen as Fine-Grained Password Policies, que permiten aplicar políticas de contrasinais diferentes a determinados usuarios ou grupos. Neste bloque só se mencionan como contexto.

Non debemos asumir que unha política de contrasinais diferente para Alumnado se consegue simplemente ligando unha GPO á OU Alumnado.

Poden usarse para ocultar Panel de control, limitar ferramentas, restrinxir cambios, configurar escritorio ou limitar opcións do menú Inicio.

Estas configuracións adoitan ir en User Configuration e deberían ligarse a unha OU que conteña usuarios.

OU=Alumnado └── OU=Curso1

Inclúe configuracións de seguridade do equipo, firewall, scripts de arranque, opcións locais de sistema ou instalación de software para equipos.

Adoitan ir en Computer Configuration e ligarse a unha OU con equipos.

OU=Equipos-Aulas

Para conectar automaticamente unidades de rede pódense usar scripts de inicio de sesión ou Group Policy Preferences.

  • H: para cartafol persoal;
  • P: para materiais de profesorado;
  • C1: para recursos de Curso1.

12. Security Filtering

Ademais de ligar unha GPO a unha OU ou dominio, pódese limitar quen a recibe mediante Security Filtering.

Condición 1
O usuario ou equipo está dentro do ámbito da ligazón.
Condición 2
O usuario ou equipo ten permisos Read e Apply Group Policy.

Por defecto, unha GPO adoita aplicarse a Authenticated Users dentro do seu ámbito.

GPO ligada a OU=Alumnado Security Filtering: GG_Alumnado_Curso1 Resultado: só usuarios da OU Alumnado que pertenzan ao grupo
O filtrado por grupos é potente, pero se se usa demasiado pode complicar moito o diagnóstico. Sempre que sexa posible, é mellor comezar cunha estrutura de OU clara.

13. Enforced e Block inheritance

13.1. Enforced

Fai que unha GPO teña prioridade especial e que non poida ser bloqueada por herdanza nas OU inferiores. Debe usarse con moito coidado.

13.2. Block inheritance

Permite que unha OU bloquee a herdanza de GPO superiores. Pode ser útil, pero complica entender por que unha política non se aplica.

Non usar Enforced nin Block inheritance nas primeiras prácticas salvo que o obxectivo sexa precisamente estudar o seu efecto.

14. Actualización e diagnóstico de GPO no cliente

As GPO aplícanse automaticamente no arranque do sistema, no inicio de sesión e periodicamente en segundo plano. Durante unha práctica, podemos forzar e diagnosticar a aplicación con comandos.

Forzar actualización

gpupdate /force

Resumo aplicado

gpresult /r

Informe HTML

gpresult /h C:\GPO_Report.html

Resultado gráfico de políticas

15. Exemplo conceptual aplicado a cifpsauce.local

15.1. Restricións de escritorio de Curso1

Obxectivo: aplicar restricións só ao alumnado de Curso1.

  • GPO: GPO_Alumnado_Curso1_Escritorio
  • Ligazón: OU=Curso1 dentro de OU=Alumnado
  • Rama: User Configuration
  • Comprobación: iniciar sesión con alu.curso1.001, executar gpupdate /force e revisar gpresult /r.

15.2. Equipos de aula

Obxectivo: aplicar unha configuración aos equipos de aula independentemente do usuario.

  • GPO: GPO_EquiposAula_SeguridadeBasica
  • Ligazón: OU=Equipos-Aulas
  • Rama: Computer Configuration
  • Comprobación: reiniciar equipo, executar gpupdate /force e revisar gpresult /r.
Política xeral de contrasinais: normalmente revisarase na Default Domain Policy, afecta ao dominio completo e non debe modificarse sen indicación clara do profesorado. Calquera cambio debe quedar documentado.

16. Boas prácticas con GPO

  • Usar nomes descritivos e coherentes.
  • Separar GPO por finalidade.
  • Non crear GPO xigantes con moitas configuracións sen relación.
  • Probar nunha OU pequena antes de aplicar ao dominio enteiro.
  • Non modificar Default Domain Policy nin Default Domain Controllers Policy sen entender o impacto.
  • Documentar cada GPO: nome, obxectivo, ligazón, configuracións principais e ámbito.
  • Evitar abusar de Security Filtering.
  • Evitar usar Enforced e Block inheritance salvo casos xustificados.
  • Revisar sempre con gpresult se a GPO se está aplicando.
  • Ter claro se a configuración é de usuario ou de equipo antes de decidir onde ligar a GPO.

19. Ideas clave do bloque

  • As GPO permiten configurar usuarios e equipos de forma centralizada.
  • Unha GPO créase primeiro e despois debe ligarse a un sitio, dominio ou OU.
  • As GPO non se vinculan directamente a grupos; os grupos poden usarse para filtrado.
  • A orde xeral de aplicación é LSDOU: Local, Site, Domain, OU.
  • As configuracións de equipo e de usuario aplícanse a obxectos distintos.
  • Para configuracións de usuario, a GPO debe afectar a usuarios; para configuracións de equipo, a equipos.
  • A política de contrasinais do dominio require especial coidado.
  • gpupdate, gpresult e rsop.msc son ferramentas básicas para diagnosticar GPO.
  • Un bo deseño de OU facilita moito a aplicación e diagnóstico das GPO.
