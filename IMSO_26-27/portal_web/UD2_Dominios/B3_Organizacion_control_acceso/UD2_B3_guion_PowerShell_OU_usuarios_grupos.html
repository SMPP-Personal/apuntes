UD2 · Bloque 3 PowerShell OU Usuarios e grupos

Guión PowerShell – Creación de OU, usuarios e grupos

Automatización da estrutura inicial de cifpsauce.local: OU, usuarios de exemplo, grupos globais de seguridade e comprobacións finais.

Antes de executar: usa un dominio limpo ou un snapshot anterior á práctica GUI. Se os obxectos xa existen, algúns comandos devolverán erro.
OU
Organización

Créanse OU para alumnado, profesorado, administración, equipos e grupos.

Contas

Usuarios xenéricos: alu.*, prof.001 e adm.001.

GG
Grupos globais

Representan colectivos; os grupos de dominio local chegarán cos permisos NTFS.

0. Punto de partida

Este guión permite crear mediante PowerShell a estrutura inicial de organización do dominio cifpsauce.local: unidades organizativas, usuarios de exemplo e grupos globais de seguridade.

Partimos da seguinte situación:

  • Estás en dcbuho.cifpsauce.local.
  • O dominio cifpsauce.local xa está instalado e funciona correctamente.
  • Iniciaches sesión cun usuario con permisos de administración no dominio.
  • Vas executar Windows PowerShell como administrador.

Aviso importante: este guión está pensado para executarse nun dominio limpo ou nun snapshot anterior á creación manual das OU, usuarios e grupos. Se xa fixeches a actividade GUI anterior, algúns comandos poden devolver erro porque os obxectos xa existen.

Neste guión crearemos grupos globais de colectivos: alumnado, profesorado e administración. Os grupos de dominio local para permisos sobre recursos, por exemplo DL_Curso1_Modificar, crearanse máis adiante na actividade de permisos NTFS.

1. Comprobar o dominio e importar o módulo de Active Directory

Abre PowerShell como administrador e executa:

Import-Module ActiveDirectory

Get-ADDomain
Get-ADForest

Debes comprobar que:

  • Get-ADDomain devolve información do dominio cifpsauce.local.
  • Get-ADForest devolve información do bosque no que o dominio raíz é cifpsauce.local.

Se estes comandos non funcionan, revisa:

  • que estás executando PowerShell nun controlador de dominio ou equipo con ferramentas RSAT instaladas;
  • que o módulo ActiveDirectory está dispoñible;
  • que iniciaches sesión cunha conta con permisos adecuados.

2. Definir variables base

Para evitar repetir rutas LDAP longas, imos gardar o DN do dominio nunha variable:

# DN base do dominio
$dominio = "DC=cifpsauce,DC=local"

# OU principais
$ouAlumnado      = "OU=Alumnado,$dominio"
$ouCurso1        = "OU=Curso1,OU=Alumnado,$dominio"
$ouCurso2        = "OU=Curso2,OU=Alumnado,$dominio"
$ouProfesorado   = "OU=Profesorado,$dominio"
$ouAdministracion = "OU=Administracion,$dominio"
$ouEquipos       = "OU=Equipos-Aulas,$dominio"
$ouGrupos        = "OU=Grupos,$dominio"

Lembra: nun Distinguished Name, as comas e a orde importan. OU=Curso1,OU=Alumnado,DC=cifpsauce,DC=local non é o mesmo que OU=Alumnado,OU=Curso1,DC=cifpsauce,DC=local.

3. Crear a estrutura de unidades organizativas

Queremos conseguir esta árbore:

cifpsauce.local
├── OU=Alumnado
│   ├── OU=Curso1
│   └── OU=Curso2
├── OU=Profesorado
├── OU=Administracion
├── OU=Equipos-Aulas
└── OU=Grupos

Executa:

# OU de primeiro nivel
New-ADOrganizationalUnit -Name "Alumnado"       -Path $dominio -ProtectedFromAccidentalDeletion $true
New-ADOrganizationalUnit -Name "Profesorado"    -Path $dominio -ProtectedFromAccidentalDeletion $true
New-ADOrganizationalUnit -Name "Administracion" -Path $dominio -ProtectedFromAccidentalDeletion $true
New-ADOrganizationalUnit -Name "Equipos-Aulas"  -Path $dominio -ProtectedFromAccidentalDeletion $true
New-ADOrganizationalUnit -Name "Grupos"         -Path $dominio -ProtectedFromAccidentalDeletion $true

# OU de segundo nivel dentro de Alumnado
New-ADOrganizationalUnit -Name "Curso1" -Path $ouAlumnado -ProtectedFromAccidentalDeletion $true
New-ADOrganizationalUnit -Name "Curso2" -Path $ouAlumnado -ProtectedFromAccidentalDeletion $true

A opción -ProtectedFromAccidentalDeletion $true activa a protección contra borrado accidental. É unha boa práctica cando creamos OU que logo van conter usuarios, equipos ou grupos.

Comprobación

Get-ADOrganizationalUnit -Filter * -SearchBase $dominio |
  Select-Object Name, DistinguishedName |
  Sort-Object DistinguishedName

Debes ver as OU creadas co DN esperado.

4. Crear contas de usuario

Imos crear usuarios de exemplo para:

  • alumnado de Curso1;
  • alumnado de Curso2;
  • profesorado;
  • administración.

Usuarios que se crearán:

Tipo Conta OU
Alumnado Curso1 alu.curso1.001 OU=Curso1,OU=Alumnado
Alumnado Curso1 alu.curso1.002 OU=Curso1,OU=Alumnado
Alumnado Curso2 alu.curso2.001 OU=Curso2,OU=Alumnado
Profesorado prof.001 OU=Profesorado
Administración adm.001 OU=Administracion

Para profesorado empregamos nomes xenéricos como prof.001, porque un profesor ou profesora pode impartir en varios cursos. O curso non debe quedar ligado necesariamente ao nome da conta.

4.1. Definir contrasinal inicial

Executa:

# Contrasinal común de laboratorio para as contas de exemplo
# O cmdlet New-ADUser espera un SecureString
$pass = Read-Host "Introduce o contrasinal inicial para as contas de exemplo" -AsSecureString

Nun contorno real non deberías reutilizar o mesmo contrasinal para todas as contas. Aquí faise só para simplificar a práctica de laboratorio.

4.2. Crear usuarios de alumnado

# alu.curso1.001
New-ADUser `
  -Name "Alumno Curso1 001" `
  -GivenName "Alumno" `
  -Surname "Curso1 001" `
  -DisplayName "Alumno Curso1 001" `
  -SamAccountName "alu.curso1.001" `
  -UserPrincipalName "alu.curso1.001@cifpsauce.local" `
  -Path $ouCurso1 `
  -AccountPassword $pass `
  -Enabled $true `
  -ChangePasswordAtLogon $true

# alu.curso1.002
New-ADUser `
  -Name "Alumno Curso1 002" `
  -GivenName "Alumno" `
  -Surname "Curso1 002" `
  -DisplayName "Alumno Curso1 002" `
  -SamAccountName "alu.curso1.002" `
  -UserPrincipalName "alu.curso1.002@cifpsauce.local" `
  -Path $ouCurso1 `
  -AccountPassword $pass `
  -Enabled $true `
  -ChangePasswordAtLogon $true

# alu.curso2.001
New-ADUser `
  -Name "Alumno Curso2 001" `
  -GivenName "Alumno" `
  -Surname "Curso2 001" `
  -DisplayName "Alumno Curso2 001" `
  -SamAccountName "alu.curso2.001" `
  -UserPrincipalName "alu.curso2.001@cifpsauce.local" `
  -Path $ouCurso2 `
  -AccountPassword $pass `
  -Enabled $true `
  -ChangePasswordAtLogon $true

Comprobación

Get-ADUser -Filter * -SearchBase $ouCurso1 |
  Select-Object SamAccountName, DistinguishedName

Get-ADUser -Filter * -SearchBase $ouCurso2 |
  Select-Object SamAccountName, DistinguishedName

Debes comprobar que:

  • alu.curso1.001 e alu.curso1.002 están en OU=Curso1;
  • alu.curso2.001 está en OU=Curso2.

5. Crear grupos globais de seguridade

Nesta práctica crearemos grupos globais de seguridade para representar colectivos de usuarios.

Grupo Finalidade
GG_Alumnado_Curso1 Alumnado de Curso1
GG_Alumnado_Curso2 Alumnado de Curso2
GG_Profesorado Profesorado
GG_Administracion Persoal de administración

Todos se crearán en OU=Grupos.

New-ADGroup `
  -Name "GG_Alumnado_Curso1" `
  -SamAccountName "GG_Alumnado_Curso1" `
  -GroupScope Global `
  -GroupCategory Security `
  -Path $ouGrupos

New-ADGroup `
  -Name "GG_Alumnado_Curso2" `
  -SamAccountName "GG_Alumnado_Curso2" `
  -GroupScope Global `
  -GroupCategory Security `
  -Path $ouGrupos

New-ADGroup `
  -Name "GG_Profesorado" `
  -SamAccountName "GG_Profesorado" `
  -GroupScope Global `
  -GroupCategory Security `
  -Path $ouGrupos

New-ADGroup `
  -Name "GG_Administracion" `
  -SamAccountName "GG_Administracion" `
  -GroupScope Global `
  -GroupCategory Security `
  -Path $ouGrupos

Estes grupos globais representan colectivos. Máis adiante, cando configuremos permisos sobre carpetas ou recursos, empregaremos grupos de dominio local segundo a estratexia AGDLP.

Comprobación

Get-ADGroup -Filter * -SearchBase $ouGrupos |
  Select-Object SamAccountName, GroupScope, GroupCategory, DistinguishedName

Debes comprobar que os grupos existen en OU=Grupos e que son de tipo Security e ámbito Global.

6. Engadir usuarios aos grupos correspondentes

# Alumnado de Curso1
Add-ADGroupMember `
  -Identity "GG_Alumnado_Curso1" `
  -Members "alu.curso1.001", "alu.curso1.002"

# Alumnado de Curso2
Add-ADGroupMember `
  -Identity "GG_Alumnado_Curso2" `
  -Members "alu.curso2.001"

# Profesorado
Add-ADGroupMember `
  -Identity "GG_Profesorado" `
  -Members "prof.001"

# Administración
Add-ADGroupMember `
  -Identity "GG_Administracion" `
  -Members "adm.001"

Comprobación de membros

Get-ADGroupMember "GG_Alumnado_Curso1" |
  Select-Object Name, SamAccountName, objectClass

Get-ADGroupMember "GG_Alumnado_Curso2" |
  Select-Object Name, SamAccountName, objectClass

Get-ADGroupMember "GG_Profesorado" |
  Select-Object Name, SamAccountName, objectClass

Get-ADGroupMember "GG_Administracion" |
  Select-Object Name, SamAccountName, objectClass

Debes comprobar que:

  • alu.curso1.001 e alu.curso1.002 aparecen como membros de GG_Alumnado_Curso1;
  • alu.curso2.001 aparece como membro de GG_Alumnado_Curso2;
  • prof.001 aparece como membro de GG_Profesorado;
  • adm.001 aparece como membro de GG_Administracion.

7. Opcional: crear conta de equipo para PC-AULA01

O habitual é que a conta de equipo se cree automaticamente cando un cliente Windows se une ao dominio. Nesta práctica pódese crear manualmente só como demostración.

New-ADComputer `
  -Name "PC-AULA01" `
  -SamAccountName "PC-AULA01$" `
  -Path $ouEquipos `
  -Enabled $true

Comprobación

Get-ADComputer -Filter * -SearchBase $ouEquipos |
  Select-Object Name, DistinguishedName

Debes comprobar que PC-AULA01 aparece en OU=Equipos-Aulas.

8. Comprobación final completa

Executa estas comprobacións finais para ter unha visión xeral da estrutura creada.

8.1. OU creadas

Get-ADOrganizationalUnit -Filter * -SearchBase $dominio |
  Select-Object Name, DistinguishedName |
  Sort-Object DistinguishedName

8.2. Usuarios creados

Get-ADUser -Filter * -SearchBase "OU=Alumnado,$dominio" |
  Select-Object SamAccountName, DistinguishedName

Get-ADUser -Filter * -SearchBase $ouProfesorado |
  Select-Object SamAccountName, DistinguishedName

Get-ADUser -Filter * -SearchBase $ouAdministracion |
  Select-Object SamAccountName, DistinguishedName

8.3. Grupos creados

Get-ADGroup -Filter * -SearchBase $ouGrupos |
  Select-Object SamAccountName, GroupScope, GroupCategory, DistinguishedName

8.4. Membros dos grupos

Get-ADGroupMember "GG_Alumnado_Curso1" |
  Select-Object Name, SamAccountName, objectClass

Get-ADGroupMember "GG_Profesorado" |
  Select-Object Name, SamAccountName, objectClass

9. Capturas obrigatorias

Entrega ou incorpora á memoria as seguintes evidencias:

  1. Saída de Get-ADDomain.
  2. Saída de Get-ADOrganizationalUnit mostrando as OU creadas.
  3. Saída de Get-ADUser mostrando os usuarios de OU=Curso1.
  4. Saída de Get-ADGroup mostrando os grupos de OU=Grupos.
  5. Saída de Get-ADGroupMember "GG_Alumnado_Curso1".
  6. Captura de Active Directory Users and Computers coa estrutura creada.
  7. Se fixeches a parte opcional, saída de Get-ADComputer mostrando PC-AULA01.

11. Erros frecuentes

Erro Consecuencia
Executar PowerShell sen permisos de administrador Algúns comandos poden fallar por falta de permisos
Non importar o módulo ActiveDirectory Os cmdlets New-ADUser, New-ADGroup, etc. non estarán dispoñibles
Escribir mal o DN O obxecto créase noutro sitio ou o comando falla
Esquecer comas nos DN O DN non é válido
Executar o guión dúas veces Pode aparecer erro porque as OU, usuarios ou grupos xa existen
Crear grupos fóra de OU=Grupos A estrutura do dominio queda menos ordenada
Crear grupos como Distribution Non serven para permisos de seguridade
Usar prof.curso1.001 Volvemos ligar a conta do profesorado a un curso concreto
Esquecer engadir usuarios aos grupos Os grupos existen pero non teñen utilidade práctica
Non comprobar a saída dos comandos Pódense arrastrar erros sen detectalos

13. Ideas clave da práctica

  • PowerShell permite automatizar a creación de estrutura en Active Directory.
  • As OU serven para organizar obxectos e facilitar administración e GPO.
  • Os grupos globais representan colectivos de usuarios.
  • Os permisos sobre recursos traballaranse máis adiante usando grupos de dominio local.
  • É fundamental comprobar sempre a saída dos comandos.
  • Un script que funciona unha vez pode fallar se se executa de novo sobre obxectos xa existentes.
