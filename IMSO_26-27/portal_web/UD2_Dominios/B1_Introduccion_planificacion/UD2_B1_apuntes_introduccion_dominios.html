Introdución e planificación de dominios

Conceptos básicos para entender que é un dominio Active Directory, como se organiza e que decisións hai que tomar antes de instalalo.

4.1. Introdución aos dominios en Windows Server 2022

Nesta unidade imos traballar con dominios en contornos baseados en Windows Server 2022, que empregan o servizo de directorio Active Directory Domain Services (AD DS) para centralizar a administración de usuarios, equipos, grupos, políticas e recursos da rede.

Antes de instalar nada, é importante comprender os conceptos básicos: que é un dominio, como se organiza, que vantaxes ten fronte a un grupo de traballo e que elementos o forman (bosques, dominios, controladores de dominio, unidades organizativas, sitios, grupos, políticas, etc.).

Usuarios e equipos

As contas deixan de xestionarse equipo por equipo e pasan a organizarse no dominio.

Políticas

As GPO permiten aplicar configuración e seguridade de maneira centralizada.

Recursos

Carpetas, impresoras e permisos poden administrarse con grupos e criterios comúns.

4.1.1. Grupo de traballo vs dominio

Nunha pequena rede doméstica é habitual ter equipos en grupo de traballo (workgroup). Nunha rede profesional, especialmente se hai moitos usuarios e equipos, o habitual é empregar dominios.

Grupo de traballo
  • Cada equipo xestiona localmente os seus usuarios e contrasinais.
  • Non hai un servizo central de directorio que garde a información de usuarios, equipos e políticas.
  • As autorizacións a recursos compartidos (cartafoles, impresoras, etc.) configúranse en cada máquina.
  • A administración depende moito de cada equipo individual.
Dominio Active Directory
  • Existe polo menos un controlador de dominio (Domain Controller, DC) que almacena a base de datos de Active Directory do dominio.
  • Se hai varios controladores de dominio, a información replícase entre eles para manter copias coherentes.
  • Os usuarios autentícanse contra o dominio, non só contra o equipo local.
  • As políticas de seguridade e configuración (GPO) poden aplicarse de forma centralizada.
  • A administración de usuarios, equipos, grupos e permisos faise dunha maneira máis organizada.

Conceptualmente, un dominio é unha fronteira de administración e de aplicación de políticas: dentro do dominio aplícanse unha serie de regras comúns para usuarios e equipos.

Matiz técnico: En Active Directory, o dominio é unha unidade lóxica fundamental para administrar obxectos e políticas. En contornos con varios dominios, o bosque é a fronteira de seguridade máis ampla. Neste bloque traballaremos cun escenario sinxelo: un bosque cun único dominio.

4.1.2. Elementos básicos dun dominio

En Active Directory existe unha estrutura xerárquica que organiza a información de forma lóxica e administrable.

Bosque Contedor lóxico máis amplo
Dominio cifpsauce.local
Controlador de dominio dcbuho.cifpsauce.local
Bosque (forest)

Un bosque é o contedor lóxico máis amplo de Active Directory.

  • Pode conter un ou varios dominios que comparten unha configuración común de AD.
  • Comparte un esquema común, é dicir, a definición dos tipos de obxectos que poden existir no directorio.
  • Comparte unha configuración global de Active Directory.
  • Nun contorno pequeno, como un laboratorio ou un CIFP cunha estrutura sinxela, é habitual traballar con un só bosque e un só dominio.
Dominio

Un dominio é unha partición lóxica dentro do bosque que agrupa usuarios, equipos, grupos e outros obxectos, e que permite aplicar unha administración común.

  • Ten un nome DNS; por exemplo, cifpsauce.local nun laboratorio.
  • Ten un ou varios controladores de dominio que almacenan a base de datos de AD correspondente ao dominio.
  • Permite organizar usuarios, equipos, grupos, permisos e políticas.
  • Funciona como unha fronteira de administración e de aplicación de políticas dentro da estrutura de AD.
Controlador de dominio

Un controlador de dominio é un servidor Windows con AD DS instalado que:

  • Almacena a base de datos de Active Directory do dominio: usuarios, grupos, equipos, unidades organizativas, políticas, etc.
  • Responde ás peticións de autenticación dos clientes do dominio.
  • Participa na replicación con outros controladores de dominio, se existen.
  • Habitualmente tamén ofrece o servizo DNS necesario para localizar os servizos de Active Directory.
Importante: Neste material evitaremos falar de “controlador de dominio principal” para non confundir este escenario co modelo antigo de PDC/BDC. En Active Directory moderno pode haber varios controladores de dominio que replican información entre si. Neste bloque teremos un primeiro DC porque será o primeiro servidor que instalaremos, non porque sexa “principal” no sentido clásico.

4.1.3. Unidades organizativas, contas e grupos

Dentro dun dominio, os obxectos organízanse para facilitar a administración, a delegación de tarefas e a aplicación de políticas.

4.1.3.1. Unidades organizativas (Organizational Units, OU)

Unha unidade organizativa (OU) é un contedor lóxico que permite agrupar obxectos dentro dun dominio: usuarios, grupos, equipos, etc.

As OU serven principalmente para:
  • Organizar os obxectos do dominio de forma clara.
  • Delegar administración; por exemplo, permitir que unha persoa xestione só as contas da OU Alumnado.
  • Vincular GPO a unha parte concreta da estrutura do dominio, por exemplo a unha OU de alumnado ou a unha OU de equipos dunha aula.
Exemplo de estrutura OU
  • OU=Alumnado
    • OU=Curso1
    • OU=Curso2
  • OU=Profesorado
  • OU=Administracion
  • OU=Equipos-Aulas
Matiz importante: As GPO non se vinculan directamente a grupos. As GPO vincúlanse normalmente a sitios, dominios ou OU. Os grupos poden empregarse para filtrar a aplicación dunha GPO, pero iso non é o mesmo que vincular a GPO ao grupo.
Diferencia clave: Unha OU organiza obxectos e permite aplicar administración ou políticas. Un grupo serve sobre todo para xestionar permisos ou agrupar contas con algunha finalidade. Pertencer a unha OU non equivale a pertencer a un grupo.

4.1.3.2. Contas de usuario e equipo

Dentro dun dominio existen varios tipos de contas. Dous dos máis importantes son:

Contas de usuario

Representan persoas, como alumnado, profesorado ou persoal administrativo.

Poden almacenar nome de inicio de sesión, contrasinal, pertenza a grupos, perfil, cartafoles persoais, etc.

Contas de equipo

Representan equipos que forman parte do dominio, como PC de aula, portátiles ou servidores.

Poden almacenar nome do equipo, pertenza ao dominio, configuración de seguridade asociada, políticas aplicadas, etc.

4.1.3.3. Grupos

Os grupos permiten asignar permisos a varios usuarios á vez, evitando ter que configurar permisos usuario por usuario. Poden organizarse por función, departamento, curso ou tipo de acceso.

Grupos de seguridade

Úsanse para asignar permisos a recursos, como cartafoles, impresoras ou outros obxectos.

Grupos de distribución

Úsanse para listas de correo electrónico; non se empregan para asignar permisos de seguridade.

Ademais, segundo o seu ámbito, os grupos de seguridade clasifícanse en tres categorías:

Tipo de grupo Que pode conter de forma habitual Uso habitual
Global Contas e outros grupos globais do mesmo dominio Agrupar usuarios con características comúns: alumnado, profesorado, persoal dun departamento, etc.
Dominio local Contas e grupos do mesmo dominio, doutros dominios do bosque ou de dominios de confianza Asignar permisos sobre recursos dun dominio concreto, como carpetas compartidas ou impresoras.
Universal Contas e grupos de calquera dominio do bosque Útil en bosques con varios dominios. Normalmente é innecesario en redes simples cun único dominio.

Nun contorno sinxelo cun único dominio, como o que empregaremos nas prácticas iniciais, traballaremos principalmente con grupos globais para agrupar usuarios:

  • GG_Alumnado_Curso1
  • GG_Alumnado_Curso2
  • GG_Profesorado
Modelo AGDLP

Máis adiante, cando se traballen permisos sobre recursos compartidos, será útil seguir unha lóxica como esta:

Contas de usuario → Grupos globais → Grupos de dominio local → Permisos sobre recursos

AAccountsContas de usuario

Non é necesario dominar este modelo neste primeiro bloque, pero convén coñecer a idea: non se recomenda dar permisos directamente usuario por usuario, senón mediante grupos ben organizados.

4.1.4. Sitios e replicación

Nota didáctica: Esta sección presenta os conceptos de sitios e replicación de Active Directory como contexto xeral. O seu desenvolvemento detallado e a configuración práctica tratarase en bloques posteriores da UD2, cando se traballe con escenarios de múltiples controladores de dominio.

Nunha rede máis ampla pode haber sitios diferentes; por exemplo, varias sedes, edificios ou segmentos de rede conectados mediante enlaces con distinta velocidade ou custo.

4.1.4.1. Sitios de Active Directory

Un sitio de Active Directory representa unha parte da rede física formada por unha ou varias subredes IP ben conectadas.

Un sitio adoita corresponderse cunha localización física ou cunha parte da rede onde:

  • Os equipos teñen boa conectividade entre si.
  • Pode haber un ou varios controladores de dominio.
  • Interesa que os clientes usen preferentemente os DC máis próximos.

Isto permítelle a Active Directory optimizar:

  • A autenticación, para que os usuarios se conecten ao DC máis axeitado.
  • A replicación, axustando a frecuencia e o custo dos enlaces entre sitios.
  • O tráfico de rede xerado polos servizos de directorio.
4.1.4.2. Replicación entre DC

Se hai varios DC, a información de Active Directory replícase entre eles para manterse coherente: usuarios, grupos, cambios de contrasinal, modificacións de OU, etc.

  • A replicación pode estar organizada por sitios: intra-sitio e inter-sitios.
  • A replicación intra-sitio adoita producirse dentro dunha rede rápida e ben conectada.
  • A replicación inter-sitios permite controlar mellor o tráfico entre sedes ou redes conectadas por enlaces máis lentos.
  • Se un DC falla, outro DC pode seguir autenticando usuarios, sempre que a infraestrutura estea correctamente configurada.

Na UD2, inicialmente traballaremos cun só DC (dcbuho) e un só sitio para simplificar.

Nun escenario típico de centro único, adoita comezarse cun só sitio. Os sitios fanse especialmente útiles cando hai varias sedes, subredes ou enlaces con distinta calidade.

4.1.5. Nomenclatura e planificación básica dun dominio

Antes de instalar un dominio en Windows Server 2022, convén planificar ben os nomes e a estrutura básica.

Unha mala planificación inicial pode provocar problemas posteriores con DNS, autenticación, organización de obxectos, permisos ou mantemento.

4.1.5.1. Nome do dominio e integración con DNS

O nome dun dominio de Active Directory permite:

  • Identificar a organización ou contorno dentro da rede.
  • Integrarse co sistema de nomes de dominio (DNS).
  • Localizar servizos fundamentais de Active Directory, como controladores de dominio, Kerberos ou LDAP.
Na UD2 usaremos
Dominio
cifpsauce.local
Neste laboratorio empregarase .local como sufixo interno para simplificar as prácticas. Non é un dominio público de Internet.
Primeiro controlador de dominio
dcbuho.cifpsauce.local
Advertencia didáctica: En prácticas de aula usaremos cifpsauce.local por simplicidade. Nun contorno real non debería escollerse automaticamente .local. É recomendable planificar o nome DNS a partir dun dominio rexistrado pola organización, por exemplo ad.exemplo.gal ou corp.exemplo.gal, para evitar conflitos e problemas de resolución de nomes.

4.1.5.2. Estrutura inicial de OU nun centro de formación

  • OU=Alumnado
    • OU=Curso1
    • OU=Curso2
  • OU=Profesorado
  • OU=Administracion
  • OU=Equipos-Aulas
Esta estrutura permite:
  • Aplicar GPO diferentes a alumnado, profesorado, administración ou equipos de aula.
  • Delegar administración; por exemplo, permitir que unha persoa xestione só unha OU concreta.
  • Separar contas de usuario e contas de equipo para aplicar políticas máis claras.
  • Manter a estrutura do dominio ordenada desde o comezo.
Boa práctica: Non convén crear unha estrutura de OU demasiado complexa ao principio. É mellor comezar cunha organización sinxela, clara e coherente coa realidade do centro.

4.1.5.3. Exemplos de nomes de usuario e grupos

4.1.8. Ideas clave do bloque

  • Un grupo de traballo é sinxelo, pero difícil de administrar cando medra o número de equipos e usuarios.
  • Un dominio Active Directory permite centralizar usuarios, equipos, grupos, políticas e recursos.
  • Un bosque é o contedor lóxico máis amplo de AD.
  • Un dominio organiza obxectos e políticas dentro do bosque.
  • Un controlador de dominio almacena a base de datos de AD e autentica usuarios e equipos.
  • Unha OU serve para organizar obxectos, delegar administración e vincular GPO.
  • Un grupo serve principalmente para agrupar contas e asignar permisos.
  • As GPO vincúlanse normalmente a sitios, dominios ou OU, non directamente a grupos.
  • Os sitios representan subredes IP ben conectadas e axudan a optimizar autenticación e replicación.
  • A planificación inicial do nome DNS, das OU e dos grupos é fundamental para evitar problemas posteriores.
