1. Objetivos y preparación del entorno

1.1. Qué vamos a aprender

En los bloques anteriores hemos trabajado JavaScript para:

  • crear y manipular datos;
  • trabajar con objetos y arrays;
  • modificar el DOM;
  • responder a eventos;
  • validar formularios.

En este bloque aprenderemos a trabajar con operaciones asíncronas, especialmente con la comunicación entre el navegador y datos externos mediante fetch().

Al finalizar deberías saber:

  • distinguir entre código síncrono y asíncrono;
  • comprender el papel de los callbacks;
  • entender qué es una promesa y cuáles son sus estados;
  • utilizar .then(), .catch() y .finally();
  • escribir funciones async;
  • utilizar await;
  • solicitar datos con fetch();
  • interpretar respuestas JSON;
  • comprobar response.ok;
  • diferenciar errores de red, errores HTTP y errores al procesar JSON;
  • validar la estructura básica de los datos recibidos;
  • mostrar estados iniciales, de carga, éxito, ausencia de resultados y error;
  • renderizar datos externos de forma segura en el DOM;
  • introducir la estructura básica de una petición POST.

En este bloque seguimos trabajando en el navegador. No crearemos servidores ni bases de datos. Nos centraremos en el JavaScript del cliente.

1.2. Estructura mínima del proyecto

Para los ejemplos con archivos JSON locales puedes utilizar:

proyecto-fetch/
├── index.html
├── script.js
├── styles.css
└── cursos.json

1.3. Es necesario utilizar un servidor web

Las peticiones como:

fetch("cursos.json");

deben probarse normalmente desde una dirección http:// o https://.

No abras el archivo HTML directamente con una ruta similar a:

file:///C:/proyecto/index.html

Utiliza un servidor local, por ejemplo:

  • Live Server en Visual Studio Code;
  • el servidor de desarrollo proporcionado por el entorno;
  • cualquier servidor web local sencillo.

Ejemplo de dirección válida:

http://127.0.0.1:5500/index.html

Esto evita restricciones del navegador al intentar leer recursos locales mediante fetch().

2. Código síncrono, asíncrono y callbacks

2.1. Código síncrono

El código síncrono se ejecuta línea a línea y en orden.

console.log("Primero");
console.log("Segundo");
console.log("Tercero");

Salida:

Primero
Segundo
Tercero

Cada instrucción se completa antes de continuar con la siguiente.

2.2. Código asíncrono

El código asíncrono permite iniciar una operación que puede tardar y continuar atendiendo otras tareas mientras esa operación permanece pendiente.

Ejemplos:

  • solicitar datos a un servidor;
  • esperar un temporizador;
  • consultar una API;
  • leer determinados recursos;
  • esperar una respuesta externa.
console.log("Inicio");

setTimeout(() => {
  console.log("Mensaje retrasado");
}, 2000);

console.log("Fin");

Salida aproximada:

Inicio
Fin
Mensaje retrasado

La función de setTimeout() se programa para ejecutarse más adelante.

2.3. Asincronía no significa paralelismo automático

La asincronía no significa que todo el código JavaScript se ejecute simultáneamente.

De forma simplificada:

  1. JavaScript ejecuta el código disponible.
  2. Algunas operaciones quedan gestionadas por el navegador.
  3. Cuando terminan, su continuación queda preparada.
  4. Esa continuación se ejecuta cuando JavaScript puede atenderla.

No estudiaremos todavía el bucle de eventos en profundidad, pero conviene distinguir:

  • asincronía: una operación puede quedar pendiente sin bloquear toda la aplicación;
  • paralelismo: varias tareas se ejecutan realmente al mismo tiempo.

2.4. Qué es un callback

Un callback es una función que se pasa como argumento para que otra operación la ejecute en un momento determinado.

En un evento:

boton.addEventListener("click", () => {
  console.log("Has pulsado el botón");
});

En un temporizador:

setTimeout(() => {
  console.log("Han pasado 2 segundos");
}, 2000);

Los callbacks son importantes para comprender:

  • eventos;
  • temporizadores;
  • promesas;
  • operaciones asíncronas.

3. Promesas

3.1. Qué es una promesa

Una promesa (Promise) representa el resultado futuro de una operación asíncrona.

Puede encontrarse en tres estados:

Estado Significado
pending La operación todavía no ha terminado.
fulfilled La operación terminó correctamente.
rejected La operación terminó con error.

Una promesa está settled cuando ya no está pendiente, es decir, cuando está:

  • fulfilled;
  • o rejected.

Muchas APIs modernas devuelven promesas. fetch() es una de ellas.

3.2. Trabajar con .then()

.then() permite indicar qué debe ocurrir cuando la promesa se resuelve correctamente.

promesa.then((resultado) => {
  console.log(resultado);
});

También puede devolver otro valor o una nueva promesa:

promesa
  .then((resultado) => {
    return transformar(resultado);
  })
  .then((resultadoTransformado) => {
    console.log(resultadoTransformado);
  });

3.3. Capturar errores con .catch()

promesa
  .then((resultado) => {
    console.log(resultado);
  })
  .catch((error) => {
    console.error(
      "Ha ocurrido un error:",
      error
    );
  });

.catch() puede recibir:

  • un rechazo de la promesa inicial;
  • un error lanzado con throw;
  • un error producido en un .then() anterior;
  • un fallo al procesar una respuesta.

3.4. Ejecutar código final con .finally()

promesa
  .then((resultado) => {
    console.log(resultado);
  })
  .catch((error) => {
    console.error(error);
  })
  .finally(() => {
    console.log("Operación finalizada");
  });

.finally() se ejecuta tanto si la operación termina correctamente como si falla.

Resulta útil para:

  • ocultar un indicador de carga;
  • volver a activar un botón;
  • limpiar un estado temporal;
  • restaurar la interfaz.

4. Introducción a fetch()

4.1. Qué hace fetch()

fetch() permite realizar peticiones HTTP desde JavaScript.

Se utiliza para:

  • cargar un archivo JSON;
  • solicitar datos a una API;
  • enviar datos a un servidor;
  • actualizar una interfaz sin recargar toda la página.

Forma básica:

fetch("datos.json")
  .then((respuesta) => {
    return respuesta.json();
  })
  .then((datos) => {
    console.log(datos);
  })
  .catch((error) => {
    console.error(
      "No se pudieron cargar los datos:",
      error
    );
  });

En este ejemplo:

  1. fetch("datos.json") devuelve una promesa.
  2. La promesa se resuelve con un objeto Response.
  3. respuesta.json() devuelve otra promesa.
  4. La segunda promesa se resuelve con los datos JavaScript.

4.2. El objeto Response

La primera promesa de fetch() no entrega directamente los datos.

Entrega un objeto Response con información como:

  • código de estado;
  • estado correcto o incorrecto;
  • cabeceras;
  • URL;
  • métodos para leer el cuerpo.
fetch("datos.json")
  .then((respuesta) => {
    console.log(respuesta.status);
    console.log(respuesta.ok);

    return respuesta.json();
  })
  .then((datos) => {
    console.log(datos);
  });

Propiedades principales:

Propiedad Significado
respuesta.ok Vale true cuando el estado HTTP está entre 200 y 299.
respuesta.status (response.status) Código HTTP recibido, por ejemplo 200, 404 o 500.
respuesta.statusText Texto asociado al estado, cuando está disponible.
respuesta.url URL final de la respuesta.

4.3. Errores de red y errores HTTP

fetch() no rechaza automáticamente su promesa por recibir un estado HTTP como:

  • 404;
  • 403;
  • 500.

En esos casos normalmente existe una respuesta HTTP válida, pero:

respuesta.ok === false

Debemos comprobarlo:

fetch("datos.json")
  .then((respuesta) => {
    if (!respuesta.ok) {
      throw new Error(
        `Error HTTP: ${respuesta.status}`
      );
    }

    return respuesta.json();
  })
  .then((datos) => {
    console.log(datos);
  })
  .catch((error) => {
    console.error(
      "No se pudieron cargar los datos:",
      error.message
    );
  });

fetch() puede rechazar su promesa por causas como:

  • error de red;
  • URL no válida;
  • petición bloqueada;
  • restricciones de seguridad;
  • cancelación;
  • otros problemas que impiden obtener una respuesta.

El catch también recibe errores lanzados manualmente y errores al procesar el contenido.

5. JSON

5.1. Qué es JSON

JSON significa JavaScript Object Notation.

Es un formato de texto utilizado para intercambiar datos entre aplicaciones.

Ejemplo:

{
  "nombre": "Ana",
  "edad": 20,
  "matriculado": true
}

Array JSON:

[
  {
    "nombre": "Ana",
    "nota": 8
  },
  {
    "nombre": "Luis",
    "nota": 6
  }
]

5.2. Reglas básicas

En JSON:

  • las claves usan comillas dobles;
  • las cadenas usan comillas dobles;
  • no se permiten comentarios;
  • no se permiten funciones;
  • no se admite una coma después del último elemento;
  • se permite null;
  • no se permiten undefined, NaN ni Infinity.

JSON válido:

{
  "nombre": "Ana",
  "nota": null
}

JSON no válido:

{
  "nombre": "Ana",
  "nota": undefined
}

5.3. JSON.stringify() y JSON.parse()

const alumno = {
  nombre: "Ana",
  nota: 8
};

const textoJSON =
  JSON.stringify(alumno);

console.log(textoJSON);

const alumnoRecuperado =
  JSON.parse(textoJSON);

console.log(
  alumnoRecuperado.nombre
);
Método Uso
JSON.stringify() Convierte un objeto o array JavaScript en texto JSON.
JSON.parse() Convierte texto JSON en datos JavaScript.

5.4. response.json()

Dentro de una función async, normalmente utilizamos:

const datos =
  await respuesta.json();

response.json():

  • lee el cuerpo de la respuesta;
  • intenta interpretarlo como JSON;
  • devuelve una promesa;
  • puede fallar si el contenido no es JSON válido.

Ejemplo:

async function cargarDatos() {
  try {
    const respuesta =
      await fetch("datos.json");

    if (!respuesta.ok) {
      throw new Error(
        `Error HTTP: ${respuesta.status}`
      );
    }

    const datos =
      await respuesta.json();

    console.log(datos);
  } catch (error) {
    console.error(
      "Error de red, HTTP o JSON:",
      error
    );
  }
}

cargarDatos();

Recibir una respuesta HTTP correcta no garantiza que el cuerpo contenga JSON válido.

6. async, await y tratamiento de errores

6.1. Qué es una función async

Una función declarada con async siempre devuelve una promesa.

async function obtenerNumero() {
  return 10;
}

La llamada no devuelve directamente 10:

const resultado =
  obtenerNumero();

console.log(resultado);

resultado es una promesa.

Podemos obtener el valor con:

obtenerNumero()
  .then((numero) => {
    console.log(numero);
  });

o desde otra función async:

async function mostrarNumero() {
  const numero =
    await obtenerNumero();

  console.log(numero);
}

6.2. Qué hace await

async function cargarDatos() {
  const respuesta =
    await fetch("datos.json");

  const datos =
    await respuesta.json();

  console.log(datos);
}

await pausa temporalmente la continuación de la función async que lo contiene hasta que la promesa se resuelve.

No bloquea toda la página. Mientras la promesa está pendiente, el navegador puede seguir gestionando:

  • eventos;
  • repintado de la interfaz;
  • otras tareas;
  • otras operaciones asíncronas.

6.3. Dónde puede utilizarse await

Como norma de este bloque, utiliza await dentro de una función async.

Incorrecto en un script clásico:

const respuesta =
  await fetch("datos.json");

Correcto:

async function cargarDatos() {
  const respuesta =
    await fetch("datos.json");
}

Existe await en el nivel superior de los módulos, pero no será contenido principal en este documento.

6.4. try...catch

async function cargarDatos() {
  try {
    const respuesta =
      await fetch("datos.json");

    if (!respuesta.ok) {
      throw new Error(
        `Error HTTP: ${respuesta.status}`
      );
    }

    const datos =
      await respuesta.json();

    console.log(datos);
  } catch (error) {
    console.error(
      "No se pudieron cargar los datos:",
      error.message
    );
  }
}

6.5. try...catch...finally

async function cargarDatos() {
  botonCargar.disabled = true;
  estado.textContent = "Cargando...";

  try {
    const respuesta =
      await fetch("datos.json");

    if (!respuesta.ok) {
      throw new Error(
        `Error HTTP: ${respuesta.status}`
      );
    }

    const datos =
      await respuesta.json();

    renderizarDatos(datos);

    estado.textContent =
      "Datos cargados correctamente.";
  } catch (error) {
    estado.textContent =
      "No se pudieron cargar los datos.";

    console.error(error);
  } finally {
    botonCargar.disabled = false;
  }
}

El bloque finally se ejecuta tanto si la operación termina correctamente como si falla.

6.6. .then() y async/await

Las dos sintaxis son válidas.

Con .then():

fetch("datos.json")
  .then((respuesta) => {
    if (!respuesta.ok) {
      throw new Error(
        `Error HTTP: ${respuesta.status}`
      );
    }

    return respuesta.json();
  })
  .then((datos) => {
    console.log(datos);
  })
  .catch((error) => {
    console.error(error);
  });

Con async/await:

async function cargarDatos() {
  try {
    const respuesta =
      await fetch("datos.json");

    if (!respuesta.ok) {
      throw new Error(
        `Error HTTP: ${respuesta.status}`
      );
    }

    const datos =
      await respuesta.json();

    console.log(datos);
  } catch (error) {
    console.error(error);
  }
}

Para código largo, async/await suele facilitar la lectura y la separación de tareas.

7. Entorno web, mismo origen y CORS básico

7.1. Qué es un origen

De forma simplificada, un origen depende de:

  • protocolo;
  • dominio;
  • puerto.

Estas direcciones no tienen necesariamente el mismo origen:

http://localhost:5500
http://localhost:3000
https://ejemplo.com

7.2. Qué es CORS

Cuando una página solicita datos de otro origen, el navegador aplica restricciones de seguridad.

El servidor remoto debe autorizar determinados accesos mediante CORS.

Un error CORS puede aparecer aunque:

  • la API exista;
  • la URL sea correcta;
  • el servidor responda;
  • el código JavaScript parezca correcto.

CORS no se soluciona desactivando comprobaciones del navegador ni añadiendo código arbitrario en el cliente. La autorización debe proceder del servidor remoto.

En este bloque solo necesitamos reconocer el problema. CORS en profundidad queda aplazado.

7.3. Diferenciar tipos de error

Tipo Ejemplo
Error de red No existe conexión o no puede alcanzarse el servidor.
Error HTTP El servidor responde con 404, 403 o 500.
Error de JSON El contenido no puede interpretarse como JSON válido.
Error de formato El JSON es válido, pero no tiene la estructura esperada.
Error CORS El navegador bloquea el acceso a la respuesta por política de origen.

8. Renderizado y estados de interfaz

8.1. Estados recomendados

Una interfaz que carga datos debería contemplar:

  1. estado inicial;
  2. estado de carga;
  3. éxito con datos;
  4. éxito sin resultados;
  5. error.

HTML:

<button
  type="button"
  id="botonCargar"
>
  Cargar cursos
</button>

<p
  id="estado"
  aria-live="polite"
>
  Pulsa para cargar datos.
</p>

<ul
  id="listaCursos"
  aria-busy="false"
></ul>

8.2. Mostrar y finalizar la carga

function mostrarCarga() {
  estado.textContent =
    "Cargando cursos...";

  botonCargar.disabled = true;

  listaCursos.setAttribute(
    "aria-busy",
    "true"
  );

  listaCursos.replaceChildren();
}

function finalizarCarga() {
  botonCargar.disabled = false;

  listaCursos.setAttribute(
    "aria-busy",
    "false"
  );
}

8.3. Renderizar datos con nodos DOM

function renderizarCursos(cursos) {
  listaCursos.replaceChildren();

  cursos.forEach((curso) => {
    const li =
      document.createElement("li");

    li.textContent =
      `${curso.nombre} - ${curso.horas} horas`;

    listaCursos.append(li);
  });
}

Utilizamos:

  • createElement();
  • textContent;
  • append();
  • replaceChildren().

No insertamos datos externos mediante innerHTML.

8.4. Estado sin resultados

function mostrarSinResultados() {
  listaCursos.replaceChildren();

  estado.textContent =
    "No hay cursos disponibles.";
}

Un array vacío no es necesariamente un error. Puede ser una respuesta válida sin resultados.

8.5. Cargar y renderizar

async function cargarCursos() {
  mostrarCarga();

  try {
    const cursos =
      await obtenerCursos();

    if (cursos.length === 0) {
      mostrarSinResultados();
      return;
    }

    renderizarCursos(cursos);

    estado.textContent =
      "Cursos cargados correctamente.";
  } catch (error) {
    estado.textContent =
      "No se pudieron cargar los cursos.";

    console.error(error);
  } finally {
    finalizarCarga();
  }
}

9. Validar y transformar datos recibidos

9.1. JSON válido no significa estructura correcta

Una respuesta puede contener JSON válido y, aun así, no incluir los datos esperados.

Esperamos un array:

if (!Array.isArray(datos)) {
  throw new Error(
    "El formato recibido no es una lista."
  );
}

9.2. Obtener cursos

async function obtenerCursos() {
  const respuesta =
    await fetch("cursos.json");

  if (!respuesta.ok) {
    throw new Error(
      `Error HTTP: ${respuesta.status}`
    );
  }

  const datos =
    await respuesta.json();

  if (!Array.isArray(datos)) {
    throw new Error(
      "Formato de datos incorrecto."
    );
  }

  return datos;
}

9.3. Comprobar propiedades básicas

function normalizarCurso(curso) {
  return {
    nombre:
      String(curso.nombre ?? "Sin nombre"),

    horas:
      Number(curso.horas)
  };
}

Podemos validar:

function esCursoValido(curso) {
  return (
    typeof curso === "object"
    && curso !== null
    && typeof curso.nombre === "string"
    && Number.isFinite(
      Number(curso.horas)
    )
  );
}

No construiremos todavía validadores complejos, pero sí comprobaremos los tipos más importantes.

9.4. Métodos de array

const cursosLargos =
  cursos.filter(
    (curso) => curso.horas >= 30
  );

const nombres =
  cursos.map(
    (curso) => curso.nombre
  );

const totalHoras =
  cursos.reduce(
    (total, curso) =>
      total + curso.horas,
    0
  );

Aplicaciones:

  • filter(): seleccionar cursos que cumplen una condición;
  • map(): obtener o transformar propiedades;
  • reduce(): calcular un total o resumen.

10. Introducción al envío de datos con POST

10.1. Estructura general

const nuevoAlumno = {
  nombre: "Ana",
  nota: 8
};

Ejemplo conceptual:

async function enviarAlumno(alumno) {
  const respuesta =
    await fetch(
      "https://api.ejemplo.com/alumnos",
      {
        method: "POST",
        headers: {
          "Content-Type":
            "application/json"
        },
        body:
          JSON.stringify(alumno)
      }
    );

  if (!respuesta.ok) {
    throw new Error(
      `Error HTTP: ${respuesta.status}`
    );
  }

  return respuesta.json();
}

La URL es ficticia. El ejemplo muestra la estructura general de una petición POST, pero no puede ejecutarse sin un servidor real que acepte los datos.

10.2. Elementos principales

Parte Significado
method: "POST" Indica que queremos enviar datos.
headers Describe el tipo de contenido enviado.
Content-Type Indica que el cuerpo contiene JSON.
body Contiene los datos enviados.
JSON.stringify() Convierte el objeto JavaScript en texto JSON.

El servidor debe:

  • existir;
  • aceptar el método;
  • aceptar el tipo de contenido;
  • procesar el cuerpo;
  • autorizar la petición mediante CORS cuando proceda.

El envío de datos será contenido secundario en este bloque.

11. Patrón recomendado y errores frecuentes

11.1. Separar responsabilidades

Una estructura clara separa:

  • obtención;
  • comprobación HTTP;
  • lectura del cuerpo;
  • validación;
  • renderizado;
  • estados de interfaz.
async function obtenerCursos() {
  const respuesta =
    await fetch("cursos.json");

  if (!respuesta.ok) {
    throw new Error(
      `Error HTTP: ${respuesta.status}`
    );
  }

  const datos =
    await respuesta.json();

  if (!Array.isArray(datos)) {
    throw new Error(
      "Formato incorrecto."
    );
  }

  return datos;
}

async function cargarCursos() {
  mostrarCarga();

  try {
    const cursos =
      await obtenerCursos();

    if (cursos.length === 0) {
      mostrarSinResultados();
      return;
    }

    renderizarCursos(cursos);
    mostrarExito();
  } catch (error) {
    mostrarError(error);
  } finally {
    finalizarCarga();
  }
}

11.2. Error: pensar que fetch() devuelve los datos

Incorrecto:

const datos =
  fetch("datos.json");

fetch() devuelve una promesa.

11.3. Error: olvidar await en fetch()

Incorrecto:

const respuesta =
  fetch("datos.json");

Correcto:

const respuesta =
  await fetch("datos.json");

11.4. Error: olvidar await en .json()

Incorrecto:

const datos =
  respuesta.json();

Correcto dentro de una función async:

async function procesarRespuesta(
  respuesta
) {
  const datos =
    await respuesta.json();

  return datos;
}

11.5. Error: no comprobar response.ok

if (!respuesta.ok) {
  throw new Error(
    `Error HTTP: ${respuesta.status}`
  );
}

11.6. Error: utilizar await fuera de async

Incorrecto en un script clásico:

const respuesta =
  await fetch("datos.json");

Correcto:

async function cargarDatos() {
  const respuesta =
    await fetch("datos.json");
}

11.7. Error: abrir con file:///

Problema:

fetch("datos.json") puede fallar al abrir
directamente index.html.

Solución:

Utilizar un servidor web local.

11.8. Error: asumir la estructura de los datos

Problemático dentro de una función async:

const datos =
  await respuesta.json();

datos.forEach(...);

El problema no es await, sino asumir sin comprobar que datos es un array.

Mejor:

if (!Array.isArray(datos)) {
  throw new Error(
    "Formato inesperado."
  );
}

11.9. Error: confundir estado vacío con error

if (datos.length === 0) {
  mostrarSinResultados();
}

Una respuesta sin elementos puede ser válida.

11.10. Error: realizar peticiones duplicadas

Durante la carga:

botonCargar.disabled = true;

Al finalizar:

botonCargar.disabled = false;

La reactivación debe realizarse en finally.

11.11. Error: usar innerHTML con datos externos

Evita:

lista.innerHTML =
  `<li>${curso.nombre}</li>`;

Usa:

const li =
  document.createElement("li");

li.textContent =
  curso.nombre;

lista.append(li);

11.12. Error: pensar que todo fallo llega de la misma forma

Debemos distinguir:

  • red;
  • HTTP;
  • JSON;
  • formato;
  • CORS.

El mensaje para el usuario puede ser general, mientras que la consola puede contener información técnica.

12.3. JavaScript completo

const botonCargar =
  document.querySelector(
    "#botonCargar"
  );

const estado =
  document.querySelector(
    "#estado"
  );

const listaCursos =
  document.querySelector(
    "#listaCursos"
  );

function mostrarCarga() {
  estado.textContent =
    "Cargando cursos...";

  botonCargar.disabled = true;

  listaCursos.setAttribute(
    "aria-busy",
    "true"
  );

  listaCursos.replaceChildren();
}

function finalizarCarga() {
  botonCargar.disabled = false;

  listaCursos.setAttribute(
    "aria-busy",
    "false"
  );
}

function mostrarSinResultados() {
  estado.textContent =
    "No hay cursos disponibles.";

  listaCursos.replaceChildren();
}

function renderizarCursos(cursos) {
  listaCursos.replaceChildren();

  cursos.forEach((curso) => {
    const li =
      document.createElement("li");

    li.textContent =
      `${curso.nombre}: ${curso.horas} horas`;

    listaCursos.append(li);
  });
}

async function obtenerCursos() {
  const respuesta =
    await fetch("cursos.json");

  if (!respuesta.ok) {
    throw new Error(
      `Error HTTP: ${respuesta.status}`
    );
  }

  const datos =
    await respuesta.json();

  if (!Array.isArray(datos)) {
    throw new Error(
      "El formato recibido no es una lista."
    );
  }

  return datos;
}

async function cargarCursos() {
  mostrarCarga();

  try {
    const cursos =
      await obtenerCursos();

    if (cursos.length === 0) {
      mostrarSinResultados();
      return;
    }

    renderizarCursos(cursos);

    estado.textContent =
      "Cursos cargados correctamente.";
  } catch (error) {
    estado.textContent =
      "No se pudieron cargar los cursos.";

    console.error(error);
  } finally {
    finalizarCarga();
  }
}

botonCargar.addEventListener(
  "click",
  cargarCursos
);

12.5. Resumen final

  • JavaScript puede iniciar operaciones asíncronas sin bloquear toda la interfaz.
  • Un callback es una función que se ejecuta cuando otra operación lo determina.
  • Una promesa puede estar pending, fulfilled o rejected.
  • .then() procesa resultados correctos.
  • .catch() captura rechazos y errores.
  • .finally() ejecuta tareas comunes al terminar.
  • fetch() devuelve una promesa que se resuelve con un objeto Response.
  • Los estados HTTP incorrectos deben comprobarse mediante response.ok.
  • response.json() devuelve otra promesa y puede fallar.
  • Una función async siempre devuelve una promesa.
  • await pausa la función asíncrona, no toda la página.
  • Los ejemplos con JSON local deben ejecutarse desde un servidor web.
  • CORS puede impedir el acceso a respuestas de otros orígenes.
  • JSON válido no garantiza que los datos tengan la estructura esperada.
  • La interfaz debe contemplar carga, éxito, vacío y error.
  • Los datos externos deben renderizarse mediante nodos DOM y textContent.
  • POST permite enviar datos, pero requiere un servidor real que los acepte.
