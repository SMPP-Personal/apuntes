UD2_B4c. AJAX: contexto, código heredado y relación con fetch()

1. Objetivos, contexto y entorno de trabajo

1.1. Qué vamos a aprender

En los documentos anteriores de la UD2 hemos trabajado la comunicación asíncrona moderna mediante:

  • promesas;
  • async y await;
  • fetch();
  • JSON;
  • validación de respuestas;
  • renderizado dinámico en el DOM;
  • estados de carga, éxito, ausencia de resultados y error.

Este documento tiene un objetivo complementario: comprender qué significa AJAX y reconocer código heredado basado en:

  • XMLHttpRequest;
  • $.ajax();
  • callbacks;
  • métodos de jqXHR.

En este curso seguiremos utilizando como enfoque principal:

const respuesta =
  await fetch(url);

const datos =
  await respuesta.json();

No sustituiremos fetch() por XMLHttpRequest ni por $.ajax(). Estudiaremos estas APIs para poder leer, comparar y mantener código existente.

1.2. Qué significa código heredado

Llamamos código heredado a código que:

  • fue escrito con tecnologías anteriores;
  • continúa funcionando;
  • forma parte de aplicaciones mantenidas durante años;
  • aparece en plantillas, tutoriales antiguos o proyectos reales;
  • no siempre conviene reescribir de inmediato.

Reconocerlo permite:

  • localizar qué petición se realiza;
  • entender cómo se procesan los datos;
  • detectar errores;
  • mantener aplicaciones existentes;
  • modernizar fragmentos de forma controlada.

1.3. Utilizar un servidor web

Los ejemplos con archivos JSON deben ejecutarse desde un servidor local.

Dirección recomendada:

http://127.0.0.1:5500/index.html

Evita abrir directamente:

file:///C:/proyecto/index.html

Tanto fetch(), como XMLHttpRequest y $.ajax() están sujetos a:

  • políticas de mismo origen;
  • restricciones de seguridad;
  • CORS cuando se solicitan recursos de otro origen.

2. Qué es AJAX

2.1. Significado histórico

AJAX significa:

Asynchronous JavaScript and XML

El nombre es histórico. En la actualidad es mucho más frecuente trabajar con JSON que con XML.

AJAX no es:

  • una función;
  • una biblioteca;
  • una API concreta;
  • una obligación de utilizar XML.

AJAX es una denominación para un patrón de comunicación:

JavaScript inicia una petición asíncrona, recibe una respuesta y actualiza parte de la página sin recargarla por completo.

2.2. Antes y después de AJAX

En una navegación tradicional:

  1. el usuario pulsa un enlace o envía un formulario;
  2. el navegador solicita una nueva página;
  3. el servidor responde;
  4. la página completa se recarga.

Con una comunicación de tipo AJAX:

  1. el usuario realiza una acción;
  2. JavaScript inicia una petición;
  3. mientras la respuesta está pendiente, la página puede atender otros eventos;
  4. llegan los datos;
  5. JavaScript actualiza únicamente la parte necesaria del DOM.

2.3. AJAX y asincronía

Decir que una petición se realiza «en segundo plano» no significa necesariamente que JavaScript ejecute ese código en otro hilo.

De forma simplificada:

  1. JavaScript inicia la operación;
  2. el navegador gestiona la espera;
  3. cuando llega la respuesta, se programa la continuación;
  4. JavaScript procesa el resultado cuando puede atenderlo.

AJAX se relaciona con la asincronía, pero no implica paralelismo automático.

3.1. Comparación general

3.2. Idea clave

XMLHttpRequest, fetch() y $.ajax() pueden utilizarse para implementar comunicaciones asociadas al patrón AJAX.

No debemos escribir:

Es más preciso escribir:

XMLHttpRequest frente a fetch

$.ajax() frente a fetch()

4. Referencia moderna con fetch()

4.1. Carga segura de productos

async function obtenerProductos() {
  const respuesta =
    await fetch(
      "datos/productos.json"
    );

  if (!respuesta.ok) {
    throw new Error(
      `Error HTTP: ${respuesta.status}`
    );
  }

  const productos =
    await respuesta.json();

  if (!Array.isArray(productos)) {
    throw new Error(
      "El formato recibido no es una lista de productos."
    );
  }

  return productos;
}

4.2. Cargar y tratar estados

async function cargarProductos() {
  botonCargar.disabled = true;

  estado.textContent =
    "Cargando productos...";

  listaProductos.setAttribute(
    "aria-busy",
    "true"
  );

  listaProductos.replaceChildren();

  try {
    const productos =
      await obtenerProductos();

    if (productos.length === 0) {
      estado.textContent =
        "No hay productos disponibles.";

      return;
    }

    renderizarProductos(
      productos
    );

    estado.textContent =
      "Productos cargados correctamente.";
  } catch (error) {
    estado.textContent =
      "No se pudieron cargar los productos.";

    console.error(error);
  } finally {
    botonCargar.disabled = false;

    listaProductos.setAttribute(
      "aria-busy",
      "false"
    );
  }
}

4.3. Qué errores pueden aparecer

Con fetch() debemos diferenciar:

  • error de red;
  • error HTTP;
  • error al interpretar JSON;
  • formato inesperado;
  • CORS;
  • error de renderizado.

Un estado 404 o 500 no provoca por sí solo el catch. Por eso comprobamos:

if (!respuesta.ok) {
  throw new Error(
    `Error HTTP: ${respuesta.status}`
  );
}

5. Código heredado con XMLHttpRequest

5.1. Qué es XMLHttpRequest

XMLHttpRequest es una API anterior a fetch().

Sigue disponible en los navegadores, pero normalmente no será nuestra primera opción para código nuevo.

Su funcionamiento se basa principalmente en:

  • creación de un objeto;
  • preparación de la petición;
  • eventos;
  • comprobación manual del estado;
  • lectura de la respuesta;
  • envío mediante send().

5.2. Ejemplo clásico con responseText

const xhr =
  new XMLHttpRequest();

xhr.open(
  "GET",
  "datos/productos.json"
);

xhr.addEventListener(
  "load",
  () => {
    if (
      xhr.status < 200
      || xhr.status >= 300
    ) {
      console.error(
        "Error HTTP:",
        xhr.status
      );

      return;
    }

    try {
      const productos =
        JSON.parse(
          xhr.responseText
        );

      if (!Array.isArray(productos)) {
        throw new Error(
          "El formato recibido no es una lista."
        );
      }

      console.log(productos);
    } catch (error) {
      console.error(
        "No se pudo interpretar la respuesta:",
        error
      );
    }
  }
);

xhr.addEventListener(
  "error",
  () => {
    console.error(
      "Error de red."
    );
  }
);

xhr.send();

5.3. Partes principales

Parte Función
new XMLHttpRequest() Crea el objeto de petición.
open(metodo, url) Prepara la petición.
load Se ejecuta cuando llega una respuesta.
error Se ejecuta ante determinados fallos de red.
xhr.status Contiene el código HTTP.
xhr.responseText Contiene la respuesta como texto.
JSON.parse() Convierte texto JSON en datos JavaScript.
send() Envía la petición.

5.4. El evento error no detecta todos los errores

Una respuesta HTTP 404 o 500 puede llegar al evento load.

Por eso debemos comprobar:

xhr.status

El evento error no sustituye esa comprobación.

También debemos tratar por separado los errores producidos por:

JSON.parse(
  xhr.responseText
);

5.5. Petición asíncrona por defecto

Este código:

xhr.open(
  "GET",
  "datos/productos.json"
);

prepara una petición asíncrona por defecto.

No trabajaremos con peticiones XHR síncronas.

5.6. Alternativa con responseType = "json"

const xhr =
  new XMLHttpRequest();

xhr.open(
  "GET",
  "datos/productos.json"
);

xhr.responseType =
  "json";

xhr.addEventListener(
  "load",
  () => {
    if (
      xhr.status < 200
      || xhr.status >= 300
    ) {
      console.error(
        `Error HTTP: ${xhr.status}`
      );

      return;
    }

    const productos =
      xhr.response;

    if (!Array.isArray(productos)) {
      console.error(
        "Formato inesperado."
      );

      return;
    }

    console.log(productos);
  }
);

xhr.addEventListener(
  "error",
  () => {
    console.error(
      "Error de red."
    );
  }
);

xhr.send();

Este patrón evita llamar manualmente a JSON.parse(), pero conviene reconocer ambas formas.

6. Código heredado con jQuery Ajax

6.1. Requisito: versión completa de jQuery

Para utilizar:

$.ajax()

necesitamos la distribución completa de jQuery.

La versión slim no incluye los módulos Ajax.

Por tanto, una página que utilice $.ajax() no debe cargar un archivo similar a:

jquery.slim.min.js

6.2. Ejemplo con success, error y complete

$.ajax({
  url:
    "datos/productos.json",

  method:
    "GET",

  dataType:
    "json",

  success:
    function (productos) {
      if (
        !Array.isArray(productos)
      ) {
        $("#estado").text(
          "Formato de datos incorrecto."
        );

        return;
      }

      console.log(productos);
    },

  error:
    function (
      jqXHR,
      estado,
      error
    ) {
      console.error(
        jqXHR.status,
        estado,
        error
      );
    },

  complete:
    function () {
      console.log(
        "Petición finalizada."
      );
    }
});

6.3. Opciones principales de $.ajax()

Opción Significado
url Dirección del recurso.
method Método HTTP, por ejemplo "GET" o "POST".
type Nombre antiguo todavía reconocible para indicar el método.
dataType Tipo de respuesta esperado.
contentType Tipo del contenido que se envía.
data Datos enviados.
success Callback ejecutado cuando la operación termina correctamente.
error Callback ejecutado cuando jQuery considera que la petición ha fallado.
complete Callback ejecutado al finalizar, haya éxito o error.

6.4. contentType y dataType

No significan lo mismo.

contentType:
  "application/json"

indica el tipo de contenido enviado.

dataType:
  "json"

indica el tipo de respuesta que esperamos recibir.

Ejemplo:

$.ajax({
  url:
    "https://api.ejemplo.com/productos",

  method:
    "POST",

  contentType:
    "application/json",

  dataType:
    "json",

  data:
    JSON.stringify({
      nombre:
        "Teclado",

      precio:
        25
    })
});

6.5. El objeto jqXHR

$.ajax() devuelve un objeto jqXHR.

Este objeto permite encadenar:

.done()
.fail()
.always()

Ejemplo:

$.ajax({
  url:
    "datos/productos.json",

  dataType:
    "json"
})
  .done(
    function (productos) {
      console.log(
        productos
      );
    }
  )
  .fail(
    function (
      jqXHR,
      estado,
      error
    ) {
      console.error(
        jqXHR.status,
        estado,
        error
      );
    }
  )
  .always(
    function () {
      console.log(
        "Petición finalizada."
      );
    }
  );

6.6. Métodos antiguos eliminados

En código muy antiguo pueden aparecer:

jqXHR.success()
jqXHR.error()
jqXHR.complete()

Estos métodos no deben confundirse con las opciones:

success:
error:
complete:

Los métodos antiguos fueron sustituidos por:

.done()
.fail()
.always()

La URL es ficticia. El ejemplo muestra únicamente la estructura.

async function enviarProducto(
  producto
) {
  try {
    const respuesta =
      await fetch(
        "https://api.ejemplo.com/productos",
        {
          method:
            "POST",

          headers: {
            "Content-Type":
              "application/json"

      throw new Error(
        `Error HTTP: ${respuesta.status}`

      await respuesta.json();

  } catch (error) {
    console.error(
      "No se pudo enviar el producto:",

    );
  }
}

8. Diferencias en el tratamiento de errores

8.1. Comparación

Situación fetch() XMLHttpRequest $.ajax()
Error de red Rechaza la promesa Evento error Callback error o .fail()
Error HTTP Hay que comprobar response.ok Hay que comprobar xhr.status jQuery lo trata normalmente como error
JSON incorrecto Puede fallar response.json() Puede fallar JSON.parse() Puede activar error si se esperaba JSON
Formato inesperado Debe validarse manualmente Debe validarse manualmente Debe validarse manualmente
Acción final común finally Código compartido o eventos complete o .always()

8.2. No todos los errores son iguales

Debemos distinguir:

  • fallo de red;
  • error HTTP;
  • error de JSON;
  • formato inesperado;
  • CORS;
  • error de renderizado.

El mensaje para el usuario puede ser general:

No se pudieron cargar los productos.

La consola puede mostrar detalles técnicos:

console.error(error);

8.3. CORS básico

Aunque una URL exista, el navegador puede impedir que una página lea la respuesta si el servidor no autoriza el origen.

CORS no se soluciona:

  • desactivando la seguridad del navegador;
  • añadiendo código arbitrario al cliente;
  • cambiando únicamente fetch() por XHR o jQuery.

La autorización debe proceder del servidor.

9. AJAX, JSON y renderizado seguro

9.1. Patrón general

Una aplicación dinámica suele seguir este flujo:

  1. se inicia una petición;
  2. llega una respuesta;
  3. se comprueba el estado;
  4. se interpreta JSON;
  5. se valida la estructura;
  6. se trata el estado vacío;
  7. se renderizan los datos;
  8. se actualiza el estado de la interfaz;
  9. se restauran los controles.

9.2. HTML accesible

<section>
  <h2>Productos</h2>

  <p
    id="estado"
    aria-live="polite"
  >
    Pulsa para cargar los productos.
  </p>

  <button
    type="button"
    id="boton-cargar"
  >
    Cargar productos
  </button>

  <ul
    id="lista-productos"
    aria-busy="false"
  ></ul>
</section>

9.3. Renderizado seguro con DOM nativo

function renderizarProductos(
  productos
) {
  listaProductos.replaceChildren();

  productos.forEach(
    (producto) => {
      const li =
        document.createElement(
          "li"
        );

      li.textContent =
        `${producto.nombre} - ${producto.precio} €`;

      listaProductos.append(li);
    }
  );
}

Evita insertar datos externos con:

innerHTML

9.4. Renderizado seguro con jQuery

Evita:

$("#lista-productos")
  .append(
    `<li>${texto}</li>`
  );

Utiliza:

const elemento =
  $("<li>");

elemento.text(texto);

$("#lista-productos")
  .append(elemento);

El método .text() trata el contenido como texto, no como HTML.

11. Cómo leer código heredado y errores frecuentes

11.1. Estrategia de lectura

Cuando encuentres una petición heredada:

  1. localiza la URL;
  2. identifica el método HTTP;
  3. comprueba qué datos se envían;
  4. identifica el formato esperado;
  5. localiza la función de éxito;
  6. localiza el tratamiento del error;
  7. busca una acción final común;
  8. revisa cómo se modifica el DOM;
  9. comprueba si se insertan datos como HTML;
  10. determina cómo se reescribiría con fetch().

11.2. Traducir XHR a fetch()

XMLHttpRequest fetch()
new XMLHttpRequest() fetch(url)
open() URL y opciones
evento load resolución de la promesa
xhr.status response.status y response.ok
responseText response.text() o response.json()
evento error catch
send() llamada a fetch()

11.3. Error: pensar que AJAX es solo jQuery

AJAX es un concepto general.

jQuery es una de las herramientas que se ha utilizado para implementarlo.

11.4. Error: pensar que AJAX siempre usa XML

Aunque el nombre incluya XML, hoy es habitual trabajar con JSON.

11.5. Error: pensar que XHR está eliminado

XMLHttpRequest sigue disponible.

El problema no es que haya desaparecido, sino que suele resultar más verboso que fetch().

11.6. Error: no comprobar errores HTTP

Con fetch():

if (!respuesta.ok) {
  throw new Error(
    `Error HTTP: ${respuesta.status}`
  );
}

Con XHR:

if (
  xhr.status < 200
  || xhr.status >= 300
) {
  console.error(
    xhr.status
  );
}

11.7. Error: pensar que el evento error de XHR detecta un 404

Un 404 puede llegar al evento load.

Debemos revisar:

xhr.status

11.8. Error: confundir contentType y dataType

  • contentType: lo que enviamos;
  • dataType: lo que esperamos recibir.

11.9. Error: usar jQuery slim con Ajax

La versión slim no incluye:

$.ajax()

11.10. Error: utilizar métodos antiguos de jqXHR

Evita código nuevo con:

jqXHR.success()
jqXHR.error()
jqXHR.complete()

Reconoce y utiliza:

.done()
.fail()
.always()

11.11. Error: insertar datos externos como HTML

Evita:

lista.innerHTML =
  `<li>${producto.nombre}</li>`;

y:

$("#lista")
  .append(
    `<li>${producto.nombre}</li>`
  );

Utiliza nodos DOM o .text().

11.12. Error: no tratar el array vacío

if (productos.length === 0) {
  mostrarSinResultados();
}

Una respuesta vacía puede ser válida.

11.13. Error: mezclar estilos sin criterio

No conviene combinar en una misma solución:

  • XHR;
  • fetch();
  • $.ajax();
  • callbacks;
  • async/await;

salvo que el objetivo sea compararlos.

Para código nuevo en este curso utilizaremos principalmente fetch() con async/await.

12.2. Resumen final

  • AJAX es un patrón de comunicación asíncrona sin recargar toda la página.
  • No es una API ni una biblioteca concreta.
  • XMLHttpRequest es una API anterior a fetch().
  • XHR sigue disponible, pero suele ser más verboso.
  • fetch() se basa en promesas.
  • $.ajax() pertenece a jQuery.
  • $.ajax() devuelve un objeto jqXHR.
  • jQuery permite usar success, error y complete.
  • jqXHR permite encadenar .done(), .fail() y .always().
  • La versión slim de jQuery no incluye Ajax.
  • contentType describe lo enviado.
  • dataType describe la respuesta esperada.
  • Los errores de red y los errores HTTP son diferentes.
  • Los datos recibidos deben validarse.
  • Un array vacío no es necesariamente un error.
  • Los datos externos deben renderizarse como texto o mediante nodos DOM.
  • En este curso priorizamos fetch() con async/await.
